# 协作框架审核

> 每次结构性改动后必须执行。审核范围：根宪法、子宪法、Skill、MOC 结构、log、index。

## 为什么需要这个 Skill

协作框架由 30+ 个文件组成。任何一个地方的修改都会在其他文件中留下过时引用。没有强制审核机制，框架会随时间腐化。5/28 的 MOC 引导语清理导致节标题丢失、自检清单多次变化（10→9→8 项）——如果有强制审核，这些问题应在改动当时就被发现。

## 触发场景

- **结构性改动后强制触发**：log 拆分、文件夹重命名/新建、机制重写、路径变更、Skill 新增或删除
- **青城手动触发**："框架审核""审核协作机制""检查框架"
- **知识库体检后联动触发**：体检发现框架问题时

## 审核流程

### 第一步：全文件清单确认

确认以下文件全部存在且可读：

**根目录**：`CLAUDE.md`、`soul.md`、`00-对话存档/知识卡片调用记录.md`、`协作机制.md`、`我的知识树.md`

**子宪法**（5 份）：
- `01-关于我/CLAUDE.md`
- `02-知识库/CLAUDE.md`
- `00-对话存档/CLAUDE.md`
- `素材箱/CLAUDE.md`
- `07-AI产品开发/CLAUDE.md`

**Skill**：`AI协作工具/skills/` 下全部 `SKILL.md` 文件

**知识库管理文件**：`02-知识库/index.md`、`02-知识库/log.md`

**MOC 文件**（10 份）：`02-知识库/MOC/` 下全部 `.md` 文件

### 第二步：版本号一致性

逐文件提取版本号，检查：
- 根 `CLAUDE.md` 顶部版本号
- `协作机制.md` 顶部版本号
- 各 Skill 文件版本号
- 所有文件引用的 Skill 版本号是否与实际一致
- 版本号与变更记录最新条目是否匹配

### 第三步：路径引用一致性

逐文件扫描所有路径引用，验证：
- 引用的文件/文件夹是否存在
- 路径格式统一
- 编译日志→`02-知识库/log.md`，调用记录/系统变更→`00-对话存档/知识卡片调用记录.md`
- index.md 不再引用卡片平铺（检索走 MOC）

### 第四步：术语统一性

| 检查项 | 正确用语 | 废弃用语 |
|--------|---------|---------|
| 知识库定位 | 地基和数据基建 | AI 工作记忆、外脑 |
| 卡片格式 | 四区（Frontmatter→标题+正文→相关链接→来源） | 旧格式缺相关链接区 |
| 调用记录 | `00-对话存档/知识卡片调用记录.md`，累计计数，无日期列 | 旧格式含日期列 |
| MOC 格式 | `## 节标题` → `- [[卡片名]]`，无引导语无描述 | 含引导语、含 `— 描述` |
| MOC 命名 | `MOC-主题` | `主题-MOC` |
| typed link | 四标准类型：支撑/应用于/张力/互见 | 取代/互补/同源/相关/落地于等 18 种非标变体 |
| source_type | 10 种取值 | 越界值（如对话实践） |
| status | 四阶段 seed→growing→evergreen→dormant | active 等非标值 |
| 知识编译 | v3.0 三阶段框架 | v2.5 线性 9 步 |

### 第五步：网络质量引用检查

运行「知识网络质量检测」Skill。该 Skill 已覆盖结构完整度、链接密度、网络连通性、语义质量、01↔02 桥梁五个维度，包括原第五步（MOC 结构合规）和第六步（卡片格式合规）全部检查项。本 Skill 不再重复执行。

重点确认质量检测报告中以下项目：
- MOC 死链 0、空节 0
- typed link 0 越界
- source_type / status 无越界

### 第六步：index.md 准确性

- 规模数字与实际卡片数一致
- MOC 数量与实际一致
- AI 检索指引步骤完整

### 第六点五步：MOC 健康度检查

逐 MOC 扫描：

- 最近新增卡片时间（从卡片 created 字段反推）
- 最近调用时间（从调用记录反推该 MOC 下卡片的最近调用）
- 连续 30 天零增长 + 零调用 → 标 🔴，提示「该 MOC 是否该合并/归档/删除」
- 本周零增长 → 标 🟡，提示「该领域本周无新增，是否注意力收窄或该 MOC 已完成使命？」

MOC 健康度与知识卡片生命周期逻辑一致——都是「被使用才健康」。连续无活动的 MOC 不等于该删，但值得青城看一眼。输出到审核报告的 MOC 健康度一节。

### 第六点七五步：根宪法体积增长检查

读取根 CLAUDE.md 行数，与上次精简（v1.22，102 行/6.4KB）对比：

- 增长 < 10% → 正常
- 增长 10-30% → 🟡 提示，关注膨胀趋势
- 增长 > 30% → 🔴 提示「根宪法自上次精简后增长超 30%，是否需要再次精简？」触发「系统精简」Skill

### 第八步：Skill 软链接完整性

- `.claude/skills/` 下所有软链接指向的真实文件存在且可读
- `AI协作工具/skills/` 下有 SKILL.md 的目录，是否都有对应软链接

### 第九步：Skill 内部引用一致性

每个 Skill 文件内部检查：
- MOC 名称与 `02-知识库/MOC/` 下实际文件名一致
- 引用的 log 位置正确
- 引用的 index 结构描述与当前 index.md 一致
- typed link 类型描述正确（四标准类型）
- 调用记录格式描述正确（累计计数，无日期列）

### 第十步：输出审核报告

```markdown
---
created: YYYY-MM-DD
source_type: 框架审核报告
---

# 协作框架审核报告·YYYY-MM-DD

## 总体状态
[通过 / N 项待修复]

## 审核明细

### 版本号
### 路径引用
### 术语统一
### MOC 结构
### 卡片格式
### 其他

## 待修复清单

### 🔴 必须修复
### 🟡 建议修复
```

### 第十一步：修复联动

报告输出后，🔴 项默认进入修复。原则：逐项修复 → 重跑受影响步骤 → 追加 `00-对话存档/知识卡片调用记录.md` 系统变更记录。

## 铁律

- **改动即审核**：结构性改动后立即触发，不等青城说
- **全量不抽样**：所有宪法、子宪法、Skill、MOC、log、index 全量检查
- **联动不孤立**：修一处，查全部引用点
- **修复必回写**：报告 → 修复 → 重跑 → 追加日志，形成闭环
