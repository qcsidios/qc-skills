# 协作框架审核

> 版本：v2.1 · 2026-06-11

> 每次结构性改动后必须执行。审核范围：根宪法、子宪法、Skill、MOC 结构、log、index。

## 为什么需要这个 Skill

协作框架由 30+ 个文件组成。任何一个地方的修改都会在其他文件中留下过时引用。没有强制审核机制，框架会随时间腐化。5/28 的 MOC 引导语清理导致节标题丢失、自检清单多次变化（10→9→8 项）——如果有强制审核，这些问题应在改动当时就被发现。

## 触发场景

- **结构性改动后强制触发**：log 拆分、文件夹重命名/新建、机制重写、路径变更、Skill 新增或删除
- **青城手动触发**："框架审核""审核协作机制""检查框架"
- **知识库体检后联动触发**：体检发现框架问题时

## 审核流程（10 步）

### 1. 全文件清单确认

确认以下文件全部存在且可读：

- 根目录：`CLAUDE.md`、`soul.md`、`00-运转日志/知识卡片调用记录.md`
- 子宪法（10 份）：`01-关于我/`、`02-知识库/`、`00-运转日志/`、`03-认知工坊/`、`04-内容生产/`、`05-项目运营/`、`06-对外分享/`、`07-AI产品开发/`、`AI协作工具/`、`素材箱/` 下的 `CLAUDE.md`
- Skill：`AI协作工具/skills/` 下全部 `SKILL.md` 文件
- 知识库管理：`02-知识库/index.md`、`02-知识库/log.md`
- MOC：`02-知识库/MOC/` 下全部 `.md` 文件

### 2. 版本号一致性

逐文件提取版本号，检查：
- 根 `CLAUDE.md` 版本号与子宪法引用的根宪法版本号是否一致
- 各 Skill 文件版本号是否存在、是否与变更记录匹配
- 根宪法变更记录最新条目与当前版本号一致

### 3. 路径引用一致性

逐文件扫描所有路径引用，验证：
- 引用的文件/文件夹是否存在（重点：目录重命名后旧路径残留）
- 路径格式统一
- 编译日志→`02-知识库/log.md`，调用记录/系统变更→`00-运转日志/知识卡片调用记录.md`

### 4. 术语统一性

| 检查项 | 正确用语 | 废弃用语 |
|--------|---------|---------|
| 知识库定位 | 地基和数据基建 | AI 工作记忆、外脑 |
| 卡片格式 | 四区（Frontmatter→标题+正文→相关链接→来源） | 旧格式缺相关链接区 |
| 调用记录 | `00-运转日志/知识卡片调用记录.md`，累计计数，无日期列 | 旧格式含日期列 |
| MOC 格式 | `## 节标题` → `- [[卡片名]]`，无引导语无描述 | 含引导语、含 `— 描述` |
| MOC 命名 | `MOC-主题` | `主题-MOC` |
| typed link | 四标准类型：支撑/应用于/张力/互见 | 取代/互补/同源/相关等非标变体 |
| source_type | 10 种取值 | 越界值 |
| status | 四阶段 seed→growing→evergreen→dormant | active 等非标值 |
| 知识编译 | v3.0 三阶段框架 | v2.5 线性 9 步 |

### 5. Skill 软链接完整性

- `~/.claude/skills/` 和 `.claude/skills/` 下所有软链接指向的真实文件存在且可读（0 断链）
- `AI协作工具/skills/` 下有 SKILL.md 的目录，是否都有对应软链接

### 6. Skill 内部引用一致性

每个 Skill 文件内部检查：
- 路径引用是否指向存在的文件/目录
- 引用的 MOC 名称与 `02-知识库/MOC/` 下实际文件名一致
- 调用的其他 Skill 名称是否正确（未被合并或删除）
- typed link 类型描述正确（四标准类型）
- 调用记录格式描述正确（累计计数，无日期列）

### 7. index.md 与 MOC 准确性

- 规模数字与实际卡片数一致、MOC 数量与实际一致
- AI 检索指引步骤完整
- MOC 健康度检查：
  - 连续 30 天零增长 + 零调用 → 🔴 提示合并/归档
  - 本周零增长 → 🟡 提示关注

### 8. Token 审计（原系统精简）

扫描各组件的体积膨胀：

- 根宪法行数（超过 200 行时提议精简；与上次精简基准 v1.22/102 行对比，超 30% 标 🔴）
- 各 Skill 的 description 字段体积
- 协作原则中可合并或下沉的条目（超过 15 条时提议合并）
- 变更记录是否该迁移到 CHANGELOG.md（根宪法保留最近 5 条）
- 子宪法与根宪法的重复内容
- 能下沉到 Skill 的铁律不留在根宪法

### 9. 输出审核报告

写入 `00-运转日志/框架审核/YYYY-MM-DD-框架审核报告.md`（目录不存在则先创建）。

```markdown
---
created: YYYY-MM-DD
source_type: 框架审核报告
---

# 协作框架审核报告·YYYY-MM-DD

## 总体状态
[通过 / N 项待修复]

## 审核明细
### 版本号 | 路径引用 | 术语统一 | Skill 软链接 | MOC 结构 | Token 审计

## 待修复清单
### 🔴 必须修复
### 🟡 建议修复
```

### 10. 修复联动

报告输出后，🔴 项默认进入修复。原则：逐项修复 → 重跑受影响步骤 → 追加 `00-运转日志/知识卡片调用记录.md` 系统变更记录。

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 某份子宪法缺失 | 路径下找不到 `CLAUDE.md` | 标注「缺失」而不是报错退出，继续扫描其他文件 |
| 版本号不一致 | 子宪法引用的根版本号 ≠ 当前 | 标注差异，默认同步为最新版本号 |
| 软链接断链 | `readlink` 目标不存在 | 标注断链路径，提议修复或删除 |
| Token 审计超红线 | 根宪法行数 > 133 行 | 标注 🔴，提议迁移变更记录到 CHANGELOG.md |
| 审核过程中文件被修改 | git status 显示审核期间有改动 | 重新扫描受影响的检查项 |
| MOC 格式异常 | 缺 frontmatter / 标题格式不符 | 标注具体问题，不等确认不自动改 MOC 正文 |

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **只扫根宪法不扫子宪法** | 版本号/术语不一致通常藏在子宪法里 | 10 份子宪法 + 所有 Skill 全量扫描 |
| 2 | **发现问题只报告不修复** | 报告堆积 → 问题积累 → 框架腐化 | 🔴 项默认进入修复，逐项→重跑→追加日志 |
| 3 | **修复一处漏掉引用点** | 版本号更新只改了根宪法，子宪法引用旧版本号 | 修一处，grep 全局搜引用，全量同步 |
| 4 | **跳过 MOC 格式检查** | MOC 引导语残留/节标题缺失是历史高频 bug | MOC 结构检查是固定步骤，不跳过 |

## 铁律

- **改动即审核**：结构性改动后立即触发，不等青城说
- **全量不抽样**：所有宪法、子宪法、Skill、MOC、log、index 全量检查
- **联动不孤立**：修一处，查全部引用点
- **修复必回写**：报告 → 修复 → 重跑 → 追加日志，形成闭环
