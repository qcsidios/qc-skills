# typed link 审计

> 版本：v1.4 · 2026-06-27
> v1.3→v1.4：**同步知识编译 v4.0**。知识编译从三阶段重构为两阶段，链接分类步骤从 2.5 第六步变为 2.5 第四步；同步更新所有引用。

全库审计 typed link 的类型合规和语义质量。同时作为知识编译的**链接类型分类器**——编译阶段写裸链接，编译后调用本 Skill 的分类模式自动标注类型。

## 触发

- 青城说「typed link审计」「link审计」「链接审计」
- 体检联动（体检 Skill 第四步引用本 Skill 的输出）
- 大批量编译后（10+ 张新卡）AI 主动提议

## 审计流程

### 第一步：类型合规扫描

扫描全库 `**相关链接**` 节，逐条校验 typed link 前缀是否在四标准类型（支撑/互见/张力/应用于）内。常见自创类型：`支持`→支撑、`互补`→互见、`补充`→互见、`延伸`→互见、`来源`→移至 `**来源**` 行。排除正文中 `- XXX：[[` 格式的非 typed link 行。违规项直接机械修复，不等待确认。

### 第二步：张力链接全量审计

**全量检查所有张力链接对**，逐一阅读双方卡片正文，判断是否存在方向性冲突。

**张力的正确定义**：两张卡在同一情境下给出**方向相反**的指导。「照 A 做就不能照 B 做」。不是「有点不同」「角度不同」「侧重点不同」。

**审计方法**：将张力链接对分批（每批 ≤20 对），使用 Agent 工具并行分发。每对给出一条判断：✅ 确实有方向冲突 / ❌ 无冲突，应改为互见。

**常见误用模式**：
- 两张卡只是不同场景下的不同策略 → 互见
- 一张卡是另一张的延伸/补充 → 互见
- 两张卡讨论同一话题但结论不冲突 → 互见

### 第三步：支撑链接审计

支撑链接通常数量最大（1492 条），直接全量人工审计不现实。分两层：

**3a. 启发式预筛**（机械扫描，标记可疑对）：

| 可疑信号 | 判定逻辑 |
|---------|---------|
| 同源同篇 | 两张卡 `**来源**` 指向同一素材文件 → 很可能是并列关系，误标为支撑 |
| 标题高度相似 | 两张卡标题 bigram 重叠率 > 60% → 可能是一个主题的两种说法，应为互见 |
| 双向支撑 | A→B 是支撑，B→A 也是支撑 → 至少有一向是错的 |
| A 创建时间晚于 B | 被支撑的卡（A）创建时间比支撑它的卡（B）更晚 → 时序上不可能 B 在 A 上构建 |

**3b. 全量审计可疑对**：将预筛出的可疑对分批（每批 ≤20 对），使用 Agent 工具并行判断。

**支撑的正确定义**：B 在 A 的概念地基上构建。没有 A，B 的概念体系不成立。A 是地基，B 是上层建筑。

**常见误用**：
- A 和 B 是并列的不同方法 → 互见
- B 只是引用了 A 的一个概念但未在其上构建 → 互见
- A 是 B 的应用场景 → 应用于（方向反过来）

### 第四步：输出审计报告

报告写入 `00-运转日志/体检报告/YYYY-MM-DD-typed-link审计报告.md`，结构：

```markdown
# typed link 审计报告 · YYYY-MM-DD

## 总体健康度
🟢/🟡/🔴 一句话总结

## 第一步：类型合规
- 总 typed link：N
- 违规：X（已修复/待修复）
- 明细

## 第二步：张力全量审计（N 对）
- 通过：X 对
- 误标：Y 对（建议改为互见）
- 明细 + 修复状态

## 第三步：支撑审计
- 预筛可疑：M / 总1492 对
- 全量审计可疑对：通过 X，误标 Y
- 明细 + 修复状态

## 修复清单
| 文件 | 行 | 当前类型 | 目标 | 建议改为 | 状态 |
|------|-----|---------|------|---------|------|
```

## 修复

- 类型违规 → 直接机械修复
- 语义误标（张力/支撑/互见用错）→ 列候选让青城确认后批量改
- 不改卡片正文内容，只改 typed link 标签

## 铁律

- **全量审计，不抽检**：所有 typed link 必须逐对过 Agent，不采用抽检。3100 对经 155 批 Agent 并行审计验证可行
- **张力≠不同**：「说的不一样」≠张力，「照 A 做就不能照 B 做」= 张力。全库真张力约 30 对
- **支撑≠相关**：引用了 A ≠ 在 A 上构建。支撑要求 A 是 B 的概念地基。最常见的误标：把引用/同源/并列当支撑
- **互见是默认，不是偷懒**：无法明确判断支撑/张力/应用于 → 互见。但不要因为「懒得多想」就把该标支撑的标互见
- **先修复再统计**：审计中发现的违规项即时修复，报告记录修复前/修复后

## 分类模式（编译联动）

知识编译 Skill v4.0 起，编译时 `**相关链接**` 只写裸 `- [[卡片名]]`，不标类型。编译完成后调用本模式自动标注。

### 触发

- 知识编译 Skill 2.5 第四步自动调用
- 手动：对指定卡片列表运行分类

### 分类流程

**输入**：一批卡片文件名，其 `**相关链接**` 节中有裸 `- [[xxx]]`（无类型前缀）。

**分类方法**：每批 ≤20 对。对每对裸链接：
1. 读取双方卡片正文全文
2. 按四类型定义判断关系
3. 输出 `类型：[[目标]]`

**输出格式**：
```
支撑：[[xxx]]  — A是B的概念地基
互见：[[xxx]]  — 纯语义关联
张力：[[xxx]]  — 方向冲突
应用于：[[xxx]] — A方法在B场景落地
```

### 回写

分类结果直接写入卡片文件——裸 `- [[xxx]]` 替换为 `- 类型：[[xxx]]`。验证：分类后卡片中不应残留任何裸链接。

### 铁律

- **分类器只做分类**：不改卡片正文，不改链接目标，只补类型前缀
- **不判断「该不该链」**：编译时已决定链接关系，分类器只管标类型
- **歧义默认互见**：无法明确判断支撑/张力/应用于 → 互见
