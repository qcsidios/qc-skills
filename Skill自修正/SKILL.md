# Skill 自修正

> 版本：v2.1 · 2026-06-11

## 为什么需要这个机制

工具定义写得再好，也会在执行中暴露漏洞——措辞不够精确、边界条件没覆盖、反面案例缺失。每次用户纠正 AI 的执行行为，都是一次免费的工具测试。这些测试结果如果没回写到工具定义里，下次还会犯同样的错。

**覆盖范围**：所有工具，不只是 Skill。包括：
- **Skill**（SKILL.md）：定义 AI 应该怎么做事
- **Hook**（session-start.sh 等）：定义自动触发什么
- **Cron**（scheduled_tasks.json）：定义定时做什么
- **脚本**（scripts/）：定义怎么执行
- **根宪法**（CLAUDE.md）：定义协作规则
- **子宪法**（各目录 CLAUDE.md）：定义子系统规则
- **本 Skill 自身**：自修正机制本身也需要自修正（见铁律中的自修正保护条款）

**核心逻辑**：用户纠正 = 测试反馈 → 根因分析 → 修正工具 → 验证。不是靠 AI 自己计数，是靠用户的每一次纠正驱动进化。

## 触发条件

以下信号触发本 Skill：

- **用户指出 AI 执行有误**：「不对」「你理解错了」「这个字段不是这么填的」
- **用户补充 AI 遗漏的规则**：「你忘了做 X」「这个步骤你跳过了」
- **用户否定 AI 的方案并给出正确做法**：「应该这样而不是那样」
- **用户要求 AI 检查并修正**：「看看有没有问题」「把这个写进去」

**不触发的**：
- 用户对内容本身的异议（如「这段话写得不好」→ 内容问题，不是工具问题）
- 闲聊中的建议（如「以后可以试试别的方法」→ 不是纠正）
- 与工具无关的操作纠正（如「这个文件路径错了」→ 单次操作错误，不是工具定义问题）

## 流程

### 第一步：根因分析

用户纠正发生时，立即判断错误根因在哪里：

| 根因 | 处理方式 |
|------|---------|
| 工具定义不清晰/有歧义 | → 第二步：修正 |
| 工具定义缺少边界条件 | → 第二步：补充规则 |
| 工具定义有规则但 AI 没遵守 | → 不改工具，但要自问：为什么没遵守？是规则太隐蔽还是自己太粗心？ |
| AI 自己的理解偏差，工具本身没问题 | → 不改工具，但要在当前对话中修正行为 |

**判断修正哪个工具**：同一个错误可能涉及多个工具。比如「创建 Skill 没走 skill-creator」→ 根因可能在 CLAUDE.md（规则不够醒目）、也可能在具体 Skill 的流程描述里。找到最上游的根因。如果两个上游候选同等可能，优先修正更通用的工具（如 CLAUDE.md 优先于具体 Skill）。

### 第二步：提议修正

当根因在工具定义时，向用户提议具体修正方案：

> 「这次错误的根因是 XXX 里 YYY 写得不够明确。建议加一条：『ZZZ』。要改吗？」

如果用户的修正建议会引入新问题，应说明理由并提出替代方案，不盲目执行。

**修正方案优先级**：先问「这个问题能不能用脚本彻底解决？」能脚本化的不靠AI判断。

| 优先级 | 方案类型 | 适用场景 | 示例 |
|--------|---------|---------|------|
| 1 | 脚本化 | 时间判断、文件路径、格式校验等确定性逻辑 | 归档路径.sh 自动计算归档日期 |
| 2 | 规则化 | 可用 checklist 或模板约束的行为 | 存档自检清单 8 项 |
| 3 | AI判断 | 无法脚本化的主观质量问题 | 内容是否是「思考记录」而非「流水账」 |

修正类型：
- **脚本化替代**：AI判断→脚本强制执行（如「归档日计算」从规则改为脚本）
- **措辞精确化**：模糊描述→具体描述
- **补充边界条件**：只说了该怎么做→补充不该怎么做
- **新增反面案例**：只有正面示范→加反面案例
- **新增检查步骤**：缺少自检→加 checklist
- **新增铁律**：同一错误反复出现→升级为铁律
- **删除错误规则**：规则本身有问题→移除或替换

### 第三步：执行修正

用户确认后，按以下顺序执行：

1. 修正前用 `git diff` 或备份记录当前状态，便于回退
2. 用 Edit 工具精确修改目标文件（不重写整个文件）
3. 如果修正涉及根宪法 CLAUDE.md → **单独告知用户影响范围，获得确认后再改**（CLAUDE.md 影响所有会话，不同于单个 Skill 的局部修改）
4. 如果修正涉及触发机制或协作系统 → 同步检查所有调用方是否需要联动修改
5. 修正前必须先调用对应的创建/修改工具：创建 Skill → skill-creator；修改 hook → 检查 hook 脚本；修改 cron → 检查 scheduled_tasks.json。不跳过工具链直接手改

### 第四步：验证

修正完成后，在当前对话中验证修正是否生效：
- 如果当前对话还能复现场景 → 按修正后的规则重新执行，确认不再犯
- 如果当前对话无法复现 → 下次执行该工具时主动留意。如果修正后同一错误再次出现，重新从第一步开始分析，考虑之前根因判断是否正确

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 根因模糊，无法判断哪个工具 | 同一错误可能源于 CLAUDE.md 或 Skill 或 hook | 列出所有候选根因 + 各自的修正代价，让青城决定先修哪个；不因为"不确定"就不修 |
| 用户拒绝修正方案 | 用户说「不改」「不需要」 | 记录到当前对话：该纠正信号被拒绝，下次同类纠正出现时重新提议 |
| 修正后同一错误再次出现 | 已验证的修正未生效 | 重新从第一步分析——之前的根因判断可能是错的（表面修了但根因在上游） |
| 修正涉及多个工具联动 | 改 A 需要同时改 B/C/D | 一次性列出全链路改动清单，让青城看到完整影响面，逐项确认 |
| 用户纠正信号不明确 | 用户只说「不对」但没有说具体哪不对 | 先反问「具体是哪一步不符合预期？」——不确定根因不动工具 |
| 脚本化方案执行失败 | 脚本有 bug 或环境依赖缺 | 退回到规则化方案（优先级 2），暂不升级为铁律；脚本修好后再替换规则 |
| 修正目标文件被其他人同时编辑 | git status 显示冲突 | 先 stash/处理冲突，再执行修正，不覆盖他人改动 |
| 修改 Skill 自身时的保护 | 正在改 SKILL.md 时又触发新的自修正 | 🔴 排队处理：先完成当前修正→验证→再处理新的纠正信号。不同时改两个 |

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **不分析根因直接改** | 用户说「这个不对」就改当前文件，可能修的只是症状，真正的 bug 在上游 | 每次过根因分析（第一步），找最上游的修正点 |
| 2 | **自己决定怎么改不告诉用户** | AI 单方面修改工具定义，用户不知道规则变了，下次发现行为不一致 | 必须提议→确认→执行（第二步→第三步） |
| 3 | **带着错误说「已修正」但没验证** | 改完不跑验证，结果修了个寂寞 | 第四步验证必须执行，不能跳过 |
| 4 | **改一处不查联动** | 改了 Skill 不扫 CLAUDE.md 里的引用，改了 CLAUDE.md 不扫子宪法的版本号 | 联动检查是铁律，不是建议 |
| 5 | **对「不是工具问题」的纠正也强行改工具** | 内容问题/单次操作错误/闲聊建议被当成了工具缺陷，修改后反而引入噪音 | 严格对照「不触发的」清单逐项排除 |
| 6 | **绕开 skill-creator 直接手写 SKILL.md** | 创建/修改 Skill 时跳过工具链，手写的格式和约定不一致 | 永远走 skill-creator |
| 7 | **把用户的轻描淡写当纠正信号** | 「以后可以试试别的方法」不是纠正，是闲聊。强行分析会导致无意义的工具修改 | 触发的四个信号必须匹配，模糊信号先反问澄清 |
| 8 | **同一个错误反复出现不升级为铁律** | 三次复现还是用规则兜底，规则太容易被忽略 | 同一错误 ≥3 次出现 → 升级为铁律 |

## 铁律

- **用户纠正 = 自修正信号**：每次纠正都要过一遍根因分析，不放过任何一个可以改工具的机会
- **不自行修正**：根因分析后必须向用户提议，经确认才改。AI 不能单方面修改工具定义
- **修正要精确**：用 Edit 改具体行，不重写整个文件。改了什么、为什么改，要说清楚
- **联动检查**：改一个工具时不检查关联工具，遗漏的副作用会在下次执行时才暴露，修复成本翻倍
- **不跳过工具链**：创建/修改 Skill 必须走 skill-creator，修改 hook 必须检查脚本，修改 cron 必须检查配置。不绕开工具直接手改
- **不带着错误说完成**：带着错误说完成会让用户以为问题已解决，导致同样的错在生产中重现
- **自修正保护**：修改本 Skill 自身时，必须额外向用户确认当前修改内容，因为自修正机制的错误会影响所有后续修正
- **脚本优先**：修正方案优先用脚本/规则解决，AI判断是兜底。能用一个脚本彻底解决的问题，不要写一条规则让AI去记——AI会忘，脚本不会
