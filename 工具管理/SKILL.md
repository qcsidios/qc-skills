# 工具管理

> 版本：v1.1 · 2026-06-11

> 统一管理本地所有工具：Skill、MCP、CLI。解决「装了什么、装在哪、有没有重复、需不需要更新」四个问题。

## 为什么需要这个 Skill

青城的工具分散在五六个目录——`~/.claude/skills/`、项目 `.claude/skills/`、`03-认知工坊/`、`04-内容生产/`、`07-AI产品开发/`、`~/工具库/全局技能包/`——加上 MCP 配置和 npm/pip 全局包，没有人能记清全貌。这个 Skill 是工具层的「知识库体检」——统一注册表，一键清点。

## 模块一：清点与清单管理

### 触发

青城说：「看看我有哪些工具」「工具清单」「清点工具」或任何想了解全局的表述。

### 扫描路径

| 类型 | 路径 |
|------|------|
| 全局 Skill | `~/.claude/skills/`（读软链接目标，非软链接本身） |
| 项目 Skill | `.claude/skills/` |
| 领域 Skill | `03-认知工坊/skills/`、`04-内容生产/skills/`、`07-AI产品开发/skills/` |
| 中立目录 | `~/工具库/全局技能包/` |
| MCP | `~/.claude/mcp.json`、项目 `.mcp.json`、`~/工具库/全局技能包/*/mcp*.json` |
| CLI | `which` 检查常用命令（`npx` 缓存、`npm list -g --depth=0`、`pip3 list` 中的工具包） |

### 系列工具归并

清单中只保留主入口，子工具折叠。归并规则：

| 系列 | 主入口 | 子工具识别规则 | 说明 |

| taste-skill | `taste-skill` | 14 个子风格（brandkit, brutalist-skill, gpt-tasteskill, image-to-code-skill, imagegen-frontend-mobile/web, minimalist-skill, output-skill, redesign-skill, soft-skill, stitch-skill, taste-skill-v1） | 全量归属前端设计，折叠 |

| lark | `飞书 CLI` | `lark-*` 全部 Skill + `lark-cli` | 飞书办公组已移除，lark-* 作为飞书 CLI 的子模块，列表在文末 |

归并判断逻辑：同一个 GitHub 仓库安装出来的、共享同一个命名前缀的、主 SKILL.md 中声明了子 Skill 调度关系的——都算系列。

### 清单格式

写入 `AI协作工具/工具管理/工具清单.md`，**按用途分组，自建 Skill 单列**：

```markdown
# 本地工具清单

> 更新：YYYY-MM-DD · 按用途分组 · 子工具明细见文末

## 分类说明

来源：自建 | Anthropic 官方 | Claude Code 内置 | GitHub | 飞书官方 | 微信读书官方 | Get笔记官方 | 随 xxx 安装

## 一、自建 Skill（N 个）

| 名称 | 用途 | 来源 |
|------|------|------|
| 知识调用 | 回答前扫描 MOC | 自建 |
| ... | ... | ... |

## 二、开源 Skill（N 个主入口）

### 内容生产（N 主入口 + N 子工具）
### 前端设计（N）
### 开发工程（N）
### 商业诊断（N 子工具）
### 认知方法（N 子工具）
### 搜索与调研（N）
### 记忆管理（N）
### 文件处理（N）
### 系统工具（N）

## 三、MCP 工具（N 个）
## 四、CLI 工具（N 个）

## 子工具明细（全部放文末）
```

**核心原则**：自建 Skill 不分子类，一张表到底。其余工具按用途分组，统一表头 `名称 | 用途 | 来源`。正文只写主入口加括号注子工具数，完整子工具清单放文末。全量归属同一小组的父包折叠，部分归属的拆出子工具单独列、来源标注所属父包。

### 清单维护

每次操作后（安装/卸载/移动）自动更新清单，不需要青城手动触发。

## 模块二：安装与归位

### 触发

青城说：「装一下 XX」「安装这个 skill」「https://github.com/xxx 帮我装」

### 安装流程

1. **获取**：git clone 或 npx/npm/pip 安装
2. **判断归属**：

| 工具功能 | 归属目录 |
|---------|---------|
| 内容生产相关（写作/排版/发布/配图） | `04-内容生产/skills/` |
| 认知学习相关（教学/诊断/思维） | `03-认知工坊/skills/` |
| AI 产品开发相关（编程/部署/测试） | `07-AI产品开发/skills/` |
| 通用/跨领域 | `~/工具库/全局技能包/` |

3. **归位**：移动到归属目录
4. **软链接**：`~/.claude/skills/{name}` → 实际路径
5. **中立目录**：`~/工具库/全局技能包/{name}/` → 实际路径的软链接
6. **写日志**（可选）：`AI协作工具/工具管理/安装日志.md`
7. **更新清单**：调用模块一刷新

### 位置调整

如果青城后续调整目录结构，只需：
- 移动实际文件夹
- 更新软链接 `ln -sf`
- 更新清单
- 历史日志不动

## 模块三：定期体检与维护

### 触发

青城说：「工具体检」「检查工具」「看看工具有没有问题」，或每次对话结束前主动提醒（如果上次体检超过 7 天）。

### 体检项目

1. **断链检测**：`~/.claude/skills/` 下的软链接是否指向不存在的目标
2. **重复检测**：同一 Skill 是否在多个路径存在
3. **过期检测**：git clone 的 Skill，检查 `git fetch` + `git log origin/main..HEAD` 看上游有无新 commit
4. **中文缺失**：扫所有 SKILL.md 的 `description` 字段，标记中文占比 < 30% 的
5. **清单同步**：对比实际文件系统和清单，找出遗漏或多余的条目

### 更新检查

对有 git 远端的 Skill：
```bash
cd {skill_path} && git fetch --quiet 2>/dev/null && git rev-list --count HEAD..origin/main 2>/dev/null
```
输出 > 0 表示有更新。生成更新提醒列表，等青城决定是否 `git pull`。

### 输出

直接以 markdown 格式展示体检结果，无需写入文件：

## 模块四：安装评估与整合

### 触发

青城给一个 GitHub URL 或工具名，说「这个怎么样」「要不要装」「帮我看看这个」。**这是前置步骤——装之前必须先评估。**

### 评估流程

1. **扫现有库**：调模块一，拿到当前全部主入口清单
2. **读 GitHub**：用 WebFetch 获取该工具的 README/SKILL.md，提取：功能描述、技术栈、依赖、与已有工具的相似度
3. **对比分析**：逐条对比现有工具，判断：
   - **冲突**：功能高度重叠 → 列出重叠点，建议选一
   - **互补**：功能不重叠但可串联 → 给整合建议
   - **新领域**：现有工具都没覆盖 → 标注为「补缺」
4. **结合实际**：基于青城当前的工作重心（从 `01-关于我/人物档案.md` 的「当前在做」和知识库的调用热点推断），判断这个工具的实用性。不装「以后可能用」的工具
5. **输出建议**：控制在 5-8 行内，包含：一句话定论、冲突/互补分析、建议（装/不装/替代方案）

### 整合方案

当发现互补工具时，主动探讨整合可能：

**整合信号**：
- A 的输出是 B 的输入（如 guizang → 公众号发布）
- A 和 B 覆盖同一流程的不同阶段（如 taste-skill 做设计 + impeccable 做审计）
- 多个工具共享同一套配置/资源

**整合形式**：
- 轻度：写一个编排 Skill，串联调用顺序
- 中度：共享配置文件和输出目录
- 重度：合并为一个 Skill，原工具作为子模块

整合方案直接展示给青城，等确认后执行。

## 清单文件约定

所有产物写入 `AI协作工具/工具管理/`：

```
AI协作工具/工具管理/
├── SKILL.md              ← 工具管理 Skill 本体
├── 工具清单.md           ← 主清单（模块一产出）
└── 安装日志.md           ← 每次安装的记录（按需创建）
```

## 铁律

- **先评估再安装**：不给 GitHub URL 直接装，必须先跑模块四
- **清单即事实**：每次操作后更新清单，清单与实际文件系统必须一致
- **归位不散落**：新工具必须放到正确的归属目录，不放临时位置
- **全量折叠，部分拆开**：父包全部子工具属于同一小组则折叠；只有部分相关则拆出单独列，来源标注所属父包
- **中文优先**：安装后立即检查并补中文描述，不等到体检才发现
- **不装未来工具**：「以后可能用」不构成安装理由。只装当前实际需要的
- **子工具只留主入口软链接**：baoyu/dbs/taste-skill 等系列只保留主入口的软链接（如 `baoyu-skills`、`dbs`、`taste-skill`），子工具不配独立软链接。AI 调主入口后按 SKILL.md 内部路由到子目录读取子工具文件。子工具文件本体不动，只是不往 system prompt 里塞
- **本地优先于远端**：判断工具是否存在、是否需要删除——以本地实际文件和工具清单为准，不以 `.git` 目录里的 remote 地址为准。工具已脱离原始仓库独立运行，不再隶属于原项目

## 异常处理

| 场景 | 触发条件 | 处理动作 |
|------|---------|---------|
| 安装失败 | `npx skills add` / `git clone` 报错 | 告知错误信息，回退：删除已创建的空目录，不留下半成品 |
| git fetch 超时 | 体检更新检查中远端不可达 | 标注「远端不可达」而不是报错退出，跳过该工具继续下一个 |
| 软链接目标不存在 | 断链检测发现 | 列出断链路径，提议删除或修复，不等青城确认不自动清理 |
| 清单与实际不一致 | 对比发现遗漏/多余条目 | 用实际文件系统为准更新清单，告知差异项 |
| 归属判断模糊 | 工具功能跨多个领域 | 列出候选目录和依据，让青城选；默认归到最相关的那个 |
| 重复安装检测 | 同一 Skill 在多个路径存在 | 比较版本和内容，提议保留较新的、删除旧的 |
| 全局包安装路径不在中立文件夹 | npm/pip 安装后检查 | 在中立文件夹建目录、迁密钥、建软链接 |

## 反例清单（不要做的事）

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **不评估直接装** | 装完才发现功能重叠或有更好的替代品，浪费安装和清理时间 | 永远先跑模块四评估 |
| 2 | **装完不更新清单** | 清单过期 → 下次清点看不到新工具 → 重复安装 | 每次操作后立即更新清单 |
| 3 | **新工具散落在临时位置** | 记不住放哪、软链接断、重复安装 | 严格按归属表归位 |
| 4 | **清单保留已卸载工具** | 清单虚高，判断工具数量时被误导 | 卸载后立即从清单删除 |
| 5 | **装「以后可能用」的工具** | 大多数永远不会用，积累成工具债 | 只在有明确使用场景时安装 |
| 6 | **忽略中立文件夹规则** | 密钥散落、软链接缺失，出问题时找不到配置 | 安装后强制检查中立文件夹合规性 |
