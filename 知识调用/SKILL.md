# 知识调用

> 版本：v2.0 · 2026-06-11

> ⚠️ 最高优先级·前置强制 Skill。AI 每次回答前必须执行以下自检，不通过不自检就直接回答 = 违反根宪法。

## 前置自检（不可跳过）

**每次回答青城之前，先问自己两个问题：**

1. 这个回答是否涉及判断、决策、方向选择、答疑、复盘、内容创作、心智调试？
2. 如果涉及——有没有先扫描相关 MOC 和知识卡片？

**判断标准**：只要不是纯事实查询、纯工具执行确认、纯闲聊——就必须先调库。含简单判断也算。宁可多调，不可漏调。

**自检失败的处理**：如果已经开始回答了才发现没调库 → 停止回答 → 告诉青城「我先调一下知识库」→ 调库 → 基于卡片重新回答。不要硬撑着用已有知识凑合。

## 角色定位

知识库是地基和数据基建。信任层级：知识树 > 青城当下判断 > AI 输出。

- 青城的角色：定方向、做决策、反馈
- AI 的角色：执行者和辅助者，用知识库做实际工作，给出有依据的产出

调用是默认状态，不是特殊事件。不向青城汇报"我调用了哪张卡片"。

## 触发场景

所有需要判断的任务都默认触发：写内容、做决策、设计产品、答疑、复盘、选方向、心智调试。纯事实查询、纯执行任务、纯闲聊不触发。

## 调用流程

### 第一步：识别任务类型 → 定位 MOC

| 任务 | MOC |
|------|-----|
| 写内容/做 IP/选题 | `MOC-内容与传播` |
| 卖货/谈单/做发售 | `MOC-销售与转化` |
| 产品定价/服务设计 | `MOC-产品与服务` |
| 选方向/评估机会 | `MOC-战略与项目` |
| 拖延/卡住/抗内耗 | `MOC-执行力与时间` |
| 心态/自我认知/情绪 | `MOC-认知与心智` |
| 工具/AI/工作流 | `MOC-AI与工具协作` |
| 知识库本身 | `MOC-知识管理元层` |
| 商业诊断/定价/盈利 | `MOC-商业模式诊断方法论` |
| 对标分析/模仿/拆解 | `MOC-对标分析` |

跨场景任务涉及多个 MOC。

### 第二步：浏览 MOC → 定位候选卡片

MOC 格式为 `## 节标题` → `- [[卡片名]]`（无描述）。浏览时：

1. 打开第一步定位的 MOC 文件
2. 根据任务的关键概念，在对应节标题下找到候选卡片名
3. 卡片名本身就是标题——通过标题判断这张卡是否相关
4. 标题不够明确时，Read 卡片全文再判断

### 第三步：读卡 → 调用（三路径混合）

三种路径叠加使用，不是三选一。AI 内部完成，不对外输出。

**路径 1 — 结构检索（主路径）**
- 从候选卡片进入，沿 typed link（支撑 / 应用于 / 张力 / 互见）遍历相关卡片
- typed link 是有方向的语义关系，沿链接可发现跨 MOC 的关联卡片
- 适合「这个主题有哪些卡片」类问题，召回 5-10 张

**路径 2 — AI 语义召回（补盲）**
- 路径 1 召回 < 3 张，或问题措辞与卡片标题差异较大时自动叠加
- 读多个 MOC 文件的卡片列表，用模型自身语义理解挑相关卡片。必要时读全量 MOC（10 个文件可一次吃下）
- MOC 条目是裸卡片名（无描述），语义匹配依赖卡片名本身——写卡片时标题就是调用的第一入口
- 不依赖 MOC 描述做判断（MOC 已无描述），不确定时直接 Read 卡片

**路径 3 — 全文 grep（兜底）**
- 用 grep 关键词搜索 `02-知识库/知识卡片/`，匹配正文内容
- 专门处理：青城给的是引语、人名、专有名词、具体数字等字面表述
- 路径 1+2 都召不动时走这条

### 第四步：直接产出回答

基于扫描结果直接生成回答。可以自然嵌入 `[[卡片]]` 链接方便点开校验。不要写"我调用了 X 卡片"。

### 第五步：记录调用（强制）

回答完成后，必须更新调用记录。不管本次会话做了多少事、用了多少卡，会话结束前统一执行一次：

- 位置：`00-运转日志/知识卡片调用记录.md` 调用记录表
- 格式：`| [[卡片名]] | 调用次数 |`（累计计数，无日期列）
- 操作：找到对应行，次数 +1
- 判断标准：这张卡出现在了本次回答的思考或输出中 → 就算用过

本次会话中调用的 Skill 在对话存档中自然记录，无需单独维护。
- 判断标准：Skill 工具被成功调用 → 就算用过

这是固定流程，不用问青城，直接做。

## 卡片知识自下而上的路由

知识库的调用有两层路由：

**自上而下（AI 主导）**：MOC → 卡片名 → Read 全文 → 获得判断依据。这是日常调用路径，由 AI 在后台完成。

**自下而上（青城偶查）**：青城偶尔打开某张卡片，通过 typed link 跳转到相关卡片。这依赖网络互联质量——每张卡至少被另一张卡回链（入度 ≥ 1），typed link 使跳转方向明确。

两层路由的健康取决于第三阶段全量互联的质量。

## 铁律

- **默认行为**：回答前默认扫卡，是 AI 的内部肌肉记忆
- **过程透明**：青城只看到结果，不感知调用过程
- **不汇报**：不在回答里说"我调用了 X 卡片"
- **嵌入而非堆砌**：回答里自然引用 `[[卡片]]`
- **偶尔暴露引用链**：当 AI 的判断显著依赖某张卡片观点时，在回复结尾加一行脚注 `> * 主要依据：[[某卡片]]、[[某MOC]]`。频率控制——大约每 3-5 次判断出现一次，不是每次回答都加。目的不是让青城审阅调用日志，而是让知识库的存在感从"后台基础设施"变成"可感知的工作底座"
- **按名定位，按文判断**：MOC 卡片名定位 → Read 全文确认 → typed link 遍历
- **用了就记**：不设调用门槛。会话结束前统一更新 `00-运转日志/知识卡片调用记录.md` 调用记录表（每卡 +1）
