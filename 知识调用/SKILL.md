> 版本：v2.1 · 2026-06-15

> ⚠️ 最高优先级·前置强制 Skill。AI 每次回答前必须执行以下自检，不自检就直接回答 = 违反根宪法。

## 触发条件

**回答前先自问：这个回答需要依据知识 Wiki（`02-知识库/`）中的信息吗？** 需要→调库。不需要→跳过。

### 触发（需要调库）

| 场景 | 关键词 |
|------|--------|
| 涉及判断/决策/方向选择 | 「怎么选」「该不该」「先做什么」 |
| 答疑/复盘/总结 | 「为什么」「复盘一下」「之前怎么做的」 |
| 内容创作/选题 | 「写一篇」「做什么选题」「怎么表达」 |
| 心智调试/自我认知 | 「我为什么总是」「感觉自己」「怎么调整」 |
| 产品/商业/项目讨论 | 「怎么做产品」「定价」「这个生意」 |

### 不触发（不需要调库）

| 场景 | 示例 |
|------|------|
| 纯外部调研 | 搜工具、查价格、找 GitHub 项目、技术配置 |
| 纯系统维护 | 改配置、修 bug、调参数、文件操作 |
| 纯工具执行确认 | 「帮我发个消息」「查一下日程」「打开文档」 |
| 纯闲聊 | 日常寒暄、无关判断的随意对话 |

**交叉场景的判断标准**：如果青城的提问同时涉及外部信息和内部知识——比如「调研了 X 工具，结合我之前的情况选哪个」——后半句「之前的情况」触发调库，只调那部分的卡，不因有外部调研成分就全跳过。

## 角色定位

知识库是地基和数据基建。信任层级：知识树 > 青城当下判断 > AI 输出。

- 青城的角色：定方向、做决策、反馈
- AI 的角色：执行者和辅助者，用知识库做实际工作，给出有依据的产出

调用是默认状态，不是特殊事件。不向青城汇报「我调用了哪张卡片」。

## 调用流程

### 第一步：识别任务类型 → 定位 MOC

| 任务 | MOC |
|------|-----|
| 写内容/做 IP/选题 | `MOC-内容与传播` |
| 卖货/谈单/做发售 | `MOC-销售与转化` |
| 产品定价/服务设计 | `MOC-产品与服务` |
| 选方向/评估机会 | `MOC-战略与项目` |
| 拖延/卡住/抗内耗 | `MOC-执行力与时间` |
| 心态/自我认知/情绪 | `MOC-认知与心智` |
| 工具/AI/工作流 | `MOC-AI与工具协作` |
| 知识库本身 | `MOC-知识管理元层` |
| 商业诊断/定价/盈利 | `MOC-商业模式诊断方法论` |
| 对标分析/模仿/拆解 | `MOC-战略与项目` |

跨场景任务涉及多个 MOC。

### 第二步：浏览 MOC → 定位候选卡片

1. 打开第一步定位的 MOC 文件
2. 在对应节标题下找到候选卡片名（卡片名本身就是标题——通过标题判断是否相关）
3. 标题不够明确时 Read 卡片全文

### 第三步：读卡 → 调用（三路径混合）

三种路径叠加使用，不是三选一。AI 内部完成，不对外输出。

**路径 1 — 结构检索（主路径）**：从候选卡片沿 typed link（支撑/应用于/张力/互见）遍历相关卡片。适合「这个主题有哪些卡片」类问题。

**路径 2 — AI 语义召回（补盲）**：路径 1 召回不足（<3 张）或问题措辞与卡片标题差异大时，读多个 MOC 的卡片列表做语义匹配。MOC 条目是裸卡片名无描述，匹配依赖标题质量。不确定时直接 Read 卡片正文。

**路径 3 — 全文 grep（兜底）**：路径 1+2 都召不动时，用 grep 关键词搜 `02-知识库/知识卡片/` 正文。专门处理引语、人名、专有名词等字面表述。

### 第四步：直接产出回答

基于扫描结果直接生成回答。可以自然嵌入 `[[卡片]]` 链接方便点开校验。不要写「我调用了 X 卡片」。

### 调用记录

调用记录的更新不由本 Skill 直接执行。实际写盘由 cron（每 3 小时）或对话存档 Skill（条件执行——仅涉及知识 Wiki 时触发）统一处理。本 Skill 只需在回答过程中**内部分析哪些卡片被使用了**，不用显式记录。

## 卡片知识自下而上的路由

**自上而下（AI 主导）**：MOC → 卡片名 → Read 全文 → 获得判断依据。日常调用路径，AI 后台完成。

**自下而上（青城偶查）**：青城偶尔打开某张卡片，通过 typed link 跳转。这依赖网络互联质量——每张卡入度 ≥ 1。

## 铁律

- **回答前默认扫卡**：AI 的内部肌肉记忆。触发条件清单每问必过
- **过程透明，结果不透明**：青城看到带引用的结果，不感知调用过程。不汇报、不写「我调用了 X」
- **按名定位，按文判断**：MOC 卡名定位 → Read 全文确认 → typed link 遍历
- **网络互联是调用基础**：卡片入度 ≥ 1、typed link 四类型、每卡归入 MOC——这些不是编译时的形式要求，是调用能否成功的前置条件
- **不是每次回答都需要调库**：对照触发/不触发清单。纯外部调研、系统维护、闲聊——跳过比强行调库更正确

> **偶尔暴露引用链**：当 AI 的判断显著依赖某张卡片观点时，在回复结尾加一行脚注 `> * 主要依据：[[某卡片]]`。大约每 3-5 次判断出现一次，让知识库从「运转层后台」变成「可感知的工作底座」。这不是铁律，是有节制的透明度。
