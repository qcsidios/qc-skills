# 知识库体检

> 版本：v2.0 · 2026-06-11

> 当青城说「体检」「健康检查」「扫一下知识库」等，或月度自动触发时调用。

定期扫描整个知识库，识别需要维护的问题，输出体检报告供青城决策。

## 为什么需要体检

知识库是地基和数据基建，承载认知、项目管理和内容生产。地基会随时间膨胀和腐烂。新卡片在加，但旧卡片可能：链接断了、与新理解矛盾、长期不被使用、归不到现有 MOC。没有定期体检，地基会从「数据基建」变成「垃圾场」。

体检不是清理，是**诊断**。提出问题，由青城决定怎么处理。协作框架审核负责维护健康度（版本号/路径/术语），本 Skill 负责知识内容本身的健康度（链接/生命周期/调用活跃度）。

## 触发场景

- 青城手动触发（说「体检」「健康检查」等）
- 月初自动触发一次
- 大批量编译素材后（如一次入库 10+ 张卡片后）

## 体检流程

### 第一步：基础统计

扫描 `02-知识库/知识卡片/`，输出当前规模快照：卡片总数、MOC 数、最近 7 天新增。

### 第二步：元数据完整度

逐个检查每张卡片：YAML 头是否齐全（created + source_type）、是否有 `**来源**` 标注、文件名是否与 H1 标题一致。

结构完整度（MOC 覆盖率、死链）、链接密度、网络连通性、语义质量（typed link/source_type/status 越界）、01↔02 桥梁——这些快速拓扑检查由本 Skill 的第一阶段（知识网络快速定量检测，见文末）负责。本 Step 运行第一阶段并引用其输出，不重复执行。

### 第三步：概念链接深度分析

第一阶段已覆盖表面链接健康度。本步骤做更深层的概念级分析：

- **来源链接 vs 概念链接的区分**（重要）：扫描时把 `**来源**：[[xxx]]` 这行里的素材源链接独立计数，不计入"概念入度"。否则视频日记/补档/PPT 这类素材源会冒出来成假幽灵卡片，污染网络分析
- **同领域骨架卡片互链情况**：Top 10 概念入度的骨架卡片之间是否互链。同领域缺链是网络稀疏的信号

### 第四步：生命周期状态

读取根目录 `00-运转日志/知识卡片调用记录.md` 的调用记录表（格式：`| [[卡片名]] | 调用次数 |`，无日期列，累计计数），按调用次数推断生命周期：

- `active`：调用次数 ≥ 1
- `dormant`：调用次数 = 0 且入度 = 0（无调用、无人引用）
- `candidate_for_review`：调用次数 = 0 且入度 = 0 且创建时间 > 90 天（候选合并/删除，需青城确认）

若调用记录表为空（系统刚启动），回退用 `file.mtime + 入度` 估算：30 天内修改或入度 ≥ 5 → active；90 天未修改且入度 ≤ 1 → dormant 候选。

输出 dormant 候选清单时，**只提议不擅动**。经青城确认后，AI 把卡片 frontmatter 的 `status` 改为 `dormant`。dormant 卡片不删除、不从 MOC 移除——只在仪表盘里降权显示。这条机制让知识库可以无限增长而不腐烂。

#### 第四步附加：evergreen 静默检查（防核心认知过时）

evergreen 是「被反复引用的核心认知」，但核心认知也会过时。如果某张 evergreen 超过 90 天未修改且调用次数为 0，要么它是真正稳态的底层认知，要么它已经被新认知静默取代但没人发现。

- 触发条件：`status = evergreen` AND `file.mtime > 90 天`
- 输出「evergreen 静默清单」，列出卡片名 + 最后修改日 + 入度 + 调用次数
- **不擅自降级**——青城回访后确认：仍是核心则留着；已被取代则在新卡片里加 `取代：[[旧卡片]]` typed link
- 健康指标表加一行「evergreen 静默率」：90 天未动的 evergreen / evergreen 总数，超过 30% 标黄

### 第五步：状态自动升级

三级生长体系（seed → growing → evergreen）的核心问题是升级触发器缺失。绝大多数卡片卡在 seed 不是因为没有多来源验证，而是没人回头改状态。本步骤由体检自动执行，不需要人记。

#### 升级规则

| 升级 | 触发条件 | 检查方式 |
|------|---------|---------|
| seed → growing | 调用次数 ≥ 3 | 读 `00-运转日志/知识卡片调用记录.md` |
| seed → growing | 概念入度 ≥ 2（来源行不计入） | 扫描卡片入度 |
| growing → evergreen | 调用次数 ≥ 10 | 读调用记录 |
| growing → evergreen | 概念入度 ≥ 5（来源行不计入） | 扫描卡片入度 |

**入度计算规则**：排除 `**来源**：` 行中的链接和 `MOC/` 目录下的引用。只统计 `02-知识库/知识卡片/` 和 `01-关于我/` 下的概念引用。

**多来源交叉验证加成**：如果一张 seed 卡片的入链来自 ≥ 2 种不同 `source_type` 的卡片（如一张来自 视频日记、一张来自 线下分享），额外触发 seed → growing 升级。

#### 执行流程

1. **扫描候选**：遍历所有 seed 和 growing 卡片，按上表规则逐张判断
2. **列出升级清单**：输出「本次 seed → growing: N 张」「本次 growing → evergreen: M 张」
3. **自动执行 + 标记**：满足条件的卡片直接改 frontmatter `status` 字段。不满足条件的在报告中列出「差多少」（如「调用 2 次，差 1 次触发 seed→growing」）
4. **日志记录**：每次升级在 `02-知识库/log.md` 追加一行记录

#### 降级规则（补充）

- evergreen 卡片超过 90 天调用次数 = 0 → 降为 growing（不是 dormant，因为曾经是核心）
- growing 卡片调用次数 = 0 且入度 = 0 → 降为 seed

降级是保守操作——不删除、不移出 MOC，只在报告中列出。

### 第六步：语义维度扫描

结构体检查"网通不通"，语义扫描查"网密不密、有没有意义层断点"。这是 5/25 增补的核心维度。

#### 5.1 冗余 / 合并候选

文件名 bigram Jaccard ≥ 0.40 或核心短语重叠 ≥ 5 字的卡片对，列为冗余候选。**不擅自合并**——逐对让青城判断：

- 同源概念两次编译 → 改名/合并主从
- 方法论 vs 案例 → 保留两张，加 "应用于：[[ ]]" typed link
- 关键词共享但概念不同 → 保留

#### 5.2 MOC 内互链密度

逐 MOC 计算：实际边数（卡片之间互引）/ 理论最大边数（n×(n-1)/2）。

- 密度 ≥ 15%: 🟢 健康
- 10-15%: 🟡 偏疏
- < 10%: 🔴 网状度不足

密度低不等于"必须批量补链"——但如果某 MOC 持续低密度且卡数增长，说明编译时漏建链。下次新卡入库时由编译流程主动扫描旧卡找语义近邻。

#### 5.3 01↔02 跨区桥梁审查（Wiki 一体性核心）

**这是月度体检的硬性维度**。`01-关于我/` + `02-知识库/` 合起来是完整的个人 Wiki，跨区互链是网状度的关键。

- **01→02 缺口扫描**：逐个 01 文件，去掉 `[[ ]]` 内容后看正文是否裸出现 02 卡片名（全名或核心短语）。命中即缺口，建议补 `[[ ]]`
- **02→01 反向扫描**：逐个 02 卡片，看是否在正文中提到 01 文件名（"人物档案""思维方式"等）但没建链
- **健康标尺**：01 七个文件的 02 桥梁覆盖率 = 已建桥梁数 / 应建桥梁数（按高置信缺口算）≥ 80% 算健康

#### 5.4 0 入度卡片清单（网络末梢）

`concept_indegree == 0` 的卡片清单（不含 MOC、待验证）。这些卡片自己有出度但没人回链。

输出"应该被谁回链"建议——通常是同主题的高入度骨架卡片。**不擅自补链**，让青城挑出"应该成为小骨架"的，再由 AI 给源头卡片补出链。

事件性卡片（视频日记节选/补档摘要等）天然零入度合理，不强求。

#### 5.5 张力卡片显式边界审查

知识库里真正的"矛盾"很少；更常见的是**张力对**——同主题但场景不同的两张卡片，AI 调用时容易"两边都对"。

维护一份"张力对关注清单"（在体检 Skill 内或卡片 frontmatter 加 `tension_with` 字段），每次体检审查：

- 张力对之间是否互链
- 是否用 typed link `分歧边界：[[ ]]` 或 `区分维度：[[ ]]` 显式标注"什么情况用哪一张"
- 缺边界标注的对 → 输出建议补的边界表述（让青城审）

已知张力对（5/25 初识别，后续可扩充）：
- `止损` ↔ `耐心`：低成本试错止损原则 ↔ 116天出第一单的耐心参照
- `先验证方法论` ↔ `先执行`：先验证方法论再产品化 ↔ 先执行再优化
- `思考过程是财富` ↔ `行动产生信心`：思考过程是AI时代最大的财富 ↔ 行动产生信心思考不会
- `信息减负` ↔ `信息生长`：信息减负精简被动输入 ↔ 知识结构是从信息的身体上生长出来的

#### 5.6 骨架卡片互链审查

Top 10 概念入度的骨架卡片之间的互链情况。骨架之间未必都该互链（跨领域骨架本来就独立），但同领域骨架缺链是网络稀疏的信号。

输出缺链对清单，让青城挑出该补的。

### 第七步：矛盾扫描

并入 5.5 张力卡片显式边界审查。真矛盾（认知演进推翻旧卡）继续保留——找到后让青城判断：合并 / 标 `取代：[[旧卡片]]` typed link / 更新某一张。

### 第八步：索引与 MOC 同步性

MOC 覆盖率、死链、结构合规等快速检查由本 Skill 的第一阶段（知识网络快速定量检测）负责。本 Step 引用第一阶段输出，仅保留：

- 每张卡片是否至少被一个 MOC 收录？（第一阶段已覆盖，直接引用）
- index.md 提及的 MOC 是否都存在？
- 是否有重复条目？（第一阶段已覆盖）
- Skill 软链接完整性检查（`.claude/skills/` 下所有软链接是否指向有效文件）
- 知识编译 Skill 定期审查（随用随进化机制是否在运转）

### 第九步：素材箱状态

- `_待处理/` 是否非空？（应该为空）
- 各分类子文件夹是否分类正确？

### 第十步：协作系统一致性

- **Skill 软链接完整性**：扫描 `.claude/skills/` 下所有软链接，验证每个链接指向的真实文件存在且 SKILL.md 可读；检查是否有孤立软链接（指向不存在）或遗漏的 Skill（AI协作工具/skills/ 下有 SKILL.md 但没有对应软链接）
- **过时表述扫描**：grep 已废弃概念（如「AI 工作记忆」「外脑」「认知报告」），与最新定位「地基和数据基建」对齐
- **已卸插件残留**：grep `smart-connections`、`Periodic Notes`、`Linter`、`BRAT` 在协议/Skill/模板里的残留引用
- **版本号一致性**：根 `CLAUDE.md` 版本号与人物档案引用的版本号是否对齐

### 第十一步：输出体检报告

文件路径：`00-运转日志/体检报告/YYYY-MM-DD-体检报告.md`

**报告结构必须满足运转看板的抓取契约**：

- 第二级标题必须是 `## 总体X健康度`（X 可以为空、可以是「语义」「结构」「连接」等修饰词）——看板用正则 `##\s*总体[^\n]*健康度` 抓 summary
- 该标题后紧跟**空行 + 一段单行的健康度总结**（开头带 🟢/🟡/🔴 emoji 之一）——看板抓的是这一行作为卡片摘要
- 文件名格式：`YYYY-MM-DD-(主题).md`——看板用 `^\d{4}-\d{2}-\d{2}-` 识别

不满足这个契约的报告，看板会**静默跳过**不显示。如果改了标题写法或换了 emoji 必须同步检查看板抓取是否正常。

### 第十二步：报告输出后

把报告路径告诉青城。**🔴 必须处理项默认进入修复流程**（见第十步）。在 log.md 追加体检记录。

### 第十三步：修复与回写

**体检 → 修复是固定流程，不分两次对话做**：报告输出后，🔴 区的项目应当在同一对话内执行修复，除非：

- 青城明确说「先不修」或「等我看完再说」
- 修复涉及青城需要做语义/判断的决策（如「保留还是合并」「这张卡片是不是真该被回链」），此时列候选给青城裁决后再做

修复执行原则：
- 按报告里 🔴 → 🟡 → 🟢 的顺序做
- 每修一类先标 task in_progress，做完标 completed
- 批量中文写入后必须 grep 扫错字（同音/形近替换、截断乱码）
- 不在用户没问的地方"顺手优化"（外科手术式改动铁律）

修复完成后**必须回写体检报告**：

1. **健康指标表加「修复后」列**——把每个维度的修复后数值放到原数值旁边，标变化（如 26 → 21）和评级变化（🔴 → 🟢）
2. **新增「修复结果」章节**——放在「健康指标」之后、「下次体检建议」之前。包含：
   - 修复总览（动作数、涉及卡片数）
   - 各维度修复明细（每修一项列出具体改了什么）
   - **修复后未达 🟢 的项**——说明为什么没修（依赖其他流程？需要青城判断？观察期？）
   - 修复后流程沉淀（这次修复确认/扩充的流程要点）
3. **重跑机械扫描验证**——核心指标（0 入度、桥梁覆盖率、概念链接总数）必须从新数据来，不能凭印象写
4. **修复完成同步追加 00-运转日志/知识卡片调用记录.md 调用记录**——把修复也作为一条调用记下

为什么必须回写：下次体检时，「修复后」列就是这次体检的基线。没有基线就无法判断网络是在演化还是在腐烂。诊断报告不是终点，修复后的对照才是。

## 铁律

- **诊断后必修复，修复后必回写**：体检不是只出报告就完事——🔴 项默认同日修复，修复结果回写体检报告（见第十步）。下次体检的基线是上次的「修复后」列
- **简单优先**：体检要快，不做复杂图分析
- **可读优先**：报告给青城看，不要堆数据，每条都说清「问题是什么、建议怎么做、修了之后什么样」
- **修复时不擅动语义判断项**：合并/重命名/「这张卡是否该被回链」等需要语义判断的，必须列候选让青城裁决；机械补链/补字段/补 typed link 可直接做
- **不打扰**：如果一切健康，报告就一句话「健康，下次见」

# 第一阶段：知识网络快速定量检测

> 原「知识网络质量检测」Skill 已合并至此，作为体检的第一阶段执行。每新增 10 张卡片自动触发或手动触发。

## 触发

- **自动**：知识库卡片总数 ≥ 上次检测基数 + 10 时触发
- **手动**：青城说「检测知识网络」「网络质量」「链接检查」
- **体检联动**：体检流程的第一步，5 分钟内完成

## 六维定量扫描

| 维度 | 指标 | 健康线 |
|------|------|:--:|
| 入度分布 | 0 入度卡片占比 | < 20% |
| 出度分布 | 0 出度卡片占比 | < 15% |
| 桥梁卡片 | 入度 ≥ 3 的卡片数 | ≥ 总数 5% |
| 孤立节点 | 出入度均为 0 的卡片 | < 5% |
| MOC 覆盖 | 未被任何 MOC 链接的卡片 | < 30% |
| 概念链接 | typed link 总数 | ≥ 卡片数 × 0.5 |

## TypeLink 定性分析

在定量扫描基础上，抽样检查 typed link 的语义准确性——链接标签是否准确反映了两个卡片之间的关系。不准确则标记，批量修复。

## 与其他步骤的关系

- 定量检测通过 → 跳过深度体检的大部分机械扫描步骤
- 定量检测不通过 → 进入完整体检流程
- 定量检测结果作为体检报告的「健康指标」基线
