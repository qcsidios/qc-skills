## 检测流程

### 定量检测（步骤 1-5）

定量检测由脚本 `scripts/quant_check.sh` 完成，一次运行覆盖五个维度。所有脚本以知识树根目录为参数，默认自动推断。

```bash
bash scripts/quant_check.sh [知识树根目录]
```

**检测内容**：

**第一步：结构完整度** — MOC 覆盖率（每张卡片至少归入 1 个 MOC）、死链（MOC 和卡片引用全部有效）、Index 覆盖所有 MOC。合格标准：MOC 覆盖率 100%、死链 0、Index 全覆盖。

**第二步：链接密度** — 平均链接数 + ≤2 链卡片统计。合格标准：平均 ≥4.5、≤2 链 = 0。

**第三步：网络连通性** — 入度=0 卡片清单 + 每个 MOC 的跨 MOC 桥梁数。合格标准：入度=0 归零、每个 MOC 至少 2 个跨 MOC 桥梁。

**第四步：语义质量** — typed link 越界检测（只允许四标准类型）、source_type 和 status 分布（无越界值）、全部有来源标注。合格标准：typed link 0 越界、source_type 全在 10 种内、status 全在 4 种内、全部有来源。

**第五步：01↔02 桥梁** — 统计 01→02（个人说明书引用知识卡片）和 02→01（卡片引用个人说明书）的双向桥梁数。合格标准：01→02 ≥20、02→01 ≥10。

### 第六步：输出定量报告

前五步为**定量检测**——回答"骨架是否完整"。合格标准全部为硬指标（0 容忍）。基于 `quant_check.sh` 的输出整理为结构化报告：

```markdown
# 知识网络质量检测报告 · YYYY-MM-DD

## 定量检测（骨架）

### 结构完整度
- MOC 覆盖率：xxx/xxx
- 死链：x 条
- Index 覆盖：x/10

### 链接密度
- 平均链接：x.xx
- ≤2 链：x 张

### 网络连通性
- 入度=0：x 张
- 跨 MOC 桥梁：每个 MOC ≥ x 条

### 语义质量
- typed link 越界：x 处
- source_type 全在 10 种内
- status 全在 4 种内
- 无来源：x 张

### 01↔02 桥梁
- 01→02：x 条
- 02→01：x 条
```

### 第七步：TypeLink 质量结构分析

定量检测通过后，进入**定性分析**——回答"网络肌理是否丰富"。运行：

```bash
bash scripts/qual_check.sh [知识树根目录]
```

脚本输出 typed link 四类型统计表 + 支撑+应用占比 + 互见/支撑+应用比值 + 自动判定。

**互见占比判定**：
- ≤35%：层级结构扎实，支撑+应用为主导
- 35-50%：层级关系尚可，互见偏多但可控
- >50%：网络偏扁平，卡片知道彼此相关但层级关系不够丰富

互见是最弱的链接类型（仅表示"相关"，无方向性）。互见过半意味着 AI 调用时能找到相关卡片，但不知道谁支撑谁、谁是谁的应用——推理精度受限。

**张力链接审计**：张力链接是最难自然产生的类型。如果 < 30 条，说明网络缺乏"同一情境下方向不同"的边界标注。适当的张力链接是高阶推理的前提。

### 第八步：互联质量深度判定

结合定量（骨架）和定性（肌理），给出两层判定：

```markdown
## 定性分析（肌理）

### TypeLink 质量结构
| 类型 | 数量 | 占比 | 语义强度 |
|------|------|:--:|:--:|
| 支撑 | xxx | xx% | 强 |
| 应用于 | xxx | xx% | 强 |
| 张力 | xxx | xx% | 强 |
| 互见 | xxx | xx% | 弱 |

- 支撑+应用于占比：xx%
- 互见/(支撑+应用于)：x.x

### 互联质量判定
- 定量：✅ 全部通过（骨架合格）
- 定性：[层级扎实 / 偏扁平需改善 / 严重扁平]

### 🔴 必须修复（越界/死链/孤立）
### 🟡 建议优化（互见升级、应用于补缺）
```

## 改进与修复方法论

### 日常：使用驱动

每次调用卡片时，AI 顺带审视被调用的两张卡之间是"互见"还是"支撑/应用于"关系。该升级就升级。高强度链接自然从使用中生长出来——这是最高质量的方式，因为升级发生在上下文充分的环境里。

互见/(支撑+应用于) 比值应在数月内从当前水平逐步降低。

### 主动：聚焦语义扫描

当定性判定为"偏扁平"且青城要求主动改善时，执行聚焦三轮扫描。**不搞全量两两比对**（696 张卡 ≈ 24 万对，不可行），只扫描有明确信号的关系对。

**第一轮：互见升级审计**

扫描现有互见链接，AI 读 A 卡和 B 卡全文，判断是否该升级为支撑/应用于。优先处理"同 MOC 内的互见对"——同领域卡片最可能有层级关系。不强行升级不存在的层级。

```bash
# 找出同 MOC 内的互见对（数量远小于全量互见）
for moc in 02-知识库/MOC/*.md; do
  cards=$(grep -oP '\[\[([^]]+)\]\]' "$moc" | sed 's/\[\[//;s/\]\]//')
  # 取出这些卡片之间的互见链接，逐对读全文后判断升级
done
```

**第二轮：高入度卡出链审计**

入度 Top 20 的枢纽卡是网络骨架。检查这些卡的每条出链——typed link 类型是否正确。骨架的 typed link 质量决定整个网络的推理质量。

**第三轮：应用于补缺**

扫描 status=seed 且出链数 ≥4 的卡片，看它们引用的 evergreen/growing 卡之间是否存在"应用于"关系。种子卡落地时最容易产生原则→场景的链接。

**三轮扫描后**：重新跑第七步 TypeLink 分析，对比改善幅度。目标是支撑+应用于占比从当前水平提升 5-15 个百分点（视实际语义关系而定，不强造）。

### 不做什么

- **不批量硬加**：AI 只看卡片名加 typed link，准确率低，错链接比弱链接更坏
- **不全量两两比对**：696 张卡 ≈ 24 万对，成本不可接受
- **不强行升级**：两卡之间确实只有"相关"关系时，互见就是正确的 typed link

## 铁律

- **每增 10 张卡片跑定量**：知识库卡片总数较上次检测增长 ≥10 时自动触发前六步。运行 `bash scripts/quant_check.sh`
- **定性按周**：第七/八步在周度体检时深度分析。运行 `bash scripts/qual_check.sh`
- **0 容忍**：死链、入度=0、≤2链、越界四类问题发现即修
- **使用驱动优先**：层级关系的最佳来源是实际调用，不是批量扫描
- **聚焦不铺开**：语义扫描限定信号明确的关系对，不搞全量
