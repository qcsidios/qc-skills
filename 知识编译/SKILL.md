# 知识编译

> 版本：v4.0 · 2026-06-27
> v3.7→v4.0：**两阶段重构**。三阶段→两阶段：Phase 3（全量互联）合并入 Phase 2.5（编译后同步），消除 MOC 归入和 typed link 分类的重复执行。深度整理门槛从两张表压缩为决策链。第二遍提取禁令从 20 行压缩为 5 条核心规则。门控点从 3 个减为 2 个。384 行→~280 行，不删任何质量约束。

## 总览

知识编译的本质是把外部知识拆成「以后能在某个场景被调用」的原子单元，然后嵌入到已有知识网络中。一张卡片找不到使用场景，它就不必存在。

```
第一阶段                        第二阶段
素材预处理 → 深度整理（按需）  →  知识编译（卡片化）→ 编译后同步 → 归档
                                  ↓
                            个人信息同步（01-关于我/）
```

---

## 深度整理门槛（前置判断）

在进入第一阶段前，先判断当前素材是否需要深度整理。深度整理不是默认流程——是有触发条件的可选步骤。

**判断链**：素材是多人对话/逐字稿？→ 是 → **必须深度整理**。否 → 单篇录音 >30 分钟？→ 是 → 必须。否 → 课程/演讲/分享类？→ 是 → 必须。否 → 无 AI 预处理的原始录音？→ 是 → 必须。以上全否 → **跳过深度整理，全文通读后直接进入卡片化**（单篇信息点 ≤5 个的高密度短文、已有结构化 AI 摘要的笔记、文章/结构化内容、视频日记 ≤10 分钟均属此类）。

深度整理的目的：为知识编译提供信号密度 ~90% 的工作底稿，同时产出可对外分享的文章。

## 第一阶段：素材预处理与深度整理

### 1.1 扫描素材

扫描来源目录，列出待处理文件，评估总字数和信息密度。

### 1.2 拆分判断

先判断源文件的自然边界，再考虑字数。

**按人/按篇优先**：源文件由多个独立作者、独立分享者或独立主题的文章构成时，每人/每篇单独深度整理，不强行合并。

**字数辅助拆分**：按人/按篇后单篇仍过长时再按主题拆：

| 条件 | 处理方式 |
|------|---------|
| 单篇 < 2 万字 | 直接整理 |
| 单篇 2-5 万字 | 按主题拆为 2-3 篇 |
| 单篇 > 5 万字 | 按主题拆为多篇，每篇 1.5-2.5 万字 |

信息密度：逐字稿/闲聊类为低密度；文章/课程/方法论类为高密度。

### 1.3 深度整理

1. **逐文件全文深读**：每份源文件用 Read 工具完整读完。禁止用 bash 分块预览、禁止用摘要替代
2. **手工写出结构化整理文档**：按主题（不是发言顺序）归并方法论、框架、案例和金句。删除闲聊、重复、技术等待。保留关键原文并标注说话人。不编造原文没有的信息
3. **硬指标**：以有价值信息为判断标准，不设字数硬指标，宁短勿水；不允许脚本批量输出——逐句手写；每个章节需「案例 + 原则 + 边界」完整结构；每篇整理文档可提取 12-25 个独立知识点

**对话类素材专项规范**（录音逐字稿、线下讨论、课程演讲转写等多人对话场景），在上述通用要求基础上追加。详细操作规范见 `对话转写深度整理提示词.md`（同目录）。摘要：

- **开篇总览**：一句话来源 +「本次共 N 大板块」+ 每板块一句话提要
- **重点分层**：最有价值的洞察排第一、占最大篇幅。不因「讨论了很久」就多给篇幅——可能是在跑题
- **人物视角**：用真实人名或角色特征替代「说话人 X」
- **每核心板块带边界**：标注什么情况下不适用
- **文末金句汇总**：按主题分类陈列值得记住的判断句（💡标记）。金句是原文原话，不是 AI 概括
- **保留技术精度**：版本号、价格、API 名称、配置参数用精确数字或表格
- **整理姿势**：亲历者复盘，不是百科全书编辑。淘金——大量沙子扔掉，只留金粒

**深度整理文章存放**：在源文件同级目录下建 `深度整理/` 子文件夹。在第二阶段完成、归档原始文件（2.6）之前，不单独将整理文章移入素材箱子文件夹。整理文章是中间产物，不是最终归档物。

### 1.4 阶段门控

每批深度整理完成后，逐项显式输出：

- [ ] 该批次全部源文件已全文阅读（Read tool 逐段覆盖）→ 输出「已覆盖文件名 + 行数范围」
- [ ] 整理文章合格 → 输出「结构完整度 + 开篇总览/重点分层/人物视角/边界/金句是否满足」

**任一项不合格 → 阻断，不得进入下一阶段。**

## 第二阶段：知识编译与网络嵌入

### 2.1 第一遍提取

逐篇阅读深度整理文档（或原文，如果跳过了深度整理），提取值得在未来被调用的信息点：

- **概念**：新概念或定义
- **判断**：值得记住的判断/结论
- **方法**：可操作的方法/步骤/框架
- **经验**：具体的经验/教训/案例
- **洞见**：反直觉或深刻的观察
- **关于青城**：关于她的新认知（分流到 01-关于我/）

一个素材包含多个独立信息点时，拆成多张卡片。每张卡片只讲一个点。

### 2.1.5 第二遍提取（强制·不可跳过）

> ⚠️ **Agent 分发约束**：通过 agent 并行编译时，本段全文必须逐字嵌入 agent 提示词，禁止简化。简化本段是 2026-06-15 编译审计发现的漏卡率 30%+ 的直接原因。

第一遍完成后，回到原文逐段重读，寻找遗漏的信息点。核心规则：

1. **对照第一遍清单，逐段问自己**：这段里有没有第一遍没注意到的方法、判断、框架、案例、金句？
2. **重点排查易遗漏类型**：嵌套框架（作为「例子」出现的子方法论）、边界条件（什么情况下不适用）、具体数字/公式、金句原话、失败案例和教训
3. **禁止用 grep/全文搜索/关键词扫描/摘要重读替代逐段重读**——grep 只能命中已有认知框架内的词，会系统性漏掉框架外的洞察
4. 补充提取遗漏点，与第一遍合并去重
5. 已验证漏卡率：只提取一次会系统性遗漏约 20-30% 的可提取知识点（杨涛文章初提 10 张，二次提取补 4 张）

第二遍不是选做题。

**多篇连续编译门控**：每完成一篇的第二遍提取，必须显式声明「✅ 第二遍完成：N 张卡，补 X 张遗漏」，方可开始下一篇。全部文章提取完毕后统一做系统同步——系统同步是机械操作，逐篇做浪费，批末一次性做。但第二遍提取是认知工作，不能堆积。

### 2.2 编写卡片

按固定格式编写，格式不可变：

```markdown
---
created: YYYY-MM-DD
source_type: flomo|视频日记|口述|AI对话|课程|文章|线下分享|直播分享|微信记录|线下讨论
status: seed
---

# 标题

正文。[[双向链接]] 自然嵌入句子中。

**相关链接**：
- [[卡片名]]
- [[卡片名]]

**来源**：[[素材箱实际文件名]]
```

四个固定区：**Frontmatter → 标题+正文 → 相关链接 → 来源**。顺序不可变。

**Frontmatter 三字段**：

| 字段 | 说明 | 取值 |
|------|------|------|
| `created` | 创建日期 | YYYY-MM-DD |
| `source_type` | 来源类型 | 10 种取值（见下方） |
| `status` | 成熟度 | 新卡一律 `seed` |

`source_type` 十值：`flomo` / `视频日记` / `口述` / `AI对话` / `课程` / `文章` / `线下分享` / `直播分享` / `微信记录` / `线下讨论`。新来源类型出现时先确认再使用。

`status` 四阶段：`seed`（默认）→ `growing`（多来源验证）→ `evergreen`（核心反复引用）→ `dormant`（调用次数=0 且入度=0，体检 Skill 标记）。

**正文编写**：一个卡片一个点，用自己的话精炼转述；`[[双向链接]]` 自然嵌入句子，每张新卡至少链接一张已有卡片；中文写作中文标点；禁止修改原始文件内容。

**相关链接（两阶段流程）**：编译和类型判断拆开做。

- **编译时只写裸链接**：在 `**相关链接**` 节下以裸 `- [[卡片名]]` 列出，不标类型前缀。编译器只判断「连到谁」。
- **编译后自动分类**：由「typed link审计」Skill 的分类模式读取双方卡片正文，按四标准类型（支撑/互见/张力/应用于）输出标签，回写卡片。分类比编译时判断准确得多——分类器只做一件事，看到的是双方完整正文，一致性强。

**四标准类型速查**：
| 类型 | 一句话定义 |
|------|---------|
| 支撑 | B 在 A 的概念地基上构建——没有 A，B 不成立 |
| 应用于 | A 是抽象方法论，B 是具体落地场景 |
| 张力 | 同一情境方向相反的指导——照 A 做就不能照 B 做 |
| 互见 | 默认类型——以上都不是就用它 |

**来源链接铁律**：
- **用最终归档名**：卡片写入时预测素材在素材箱中的最终文件名，编译完成后归档立即创建，卡片链接即可解析
- **用全名，不用缩写**：禁止 `[[补档1]]`、`[[剪存-hylis]]` 等缩写。用素材箱实际文件名
- **路径可选但推荐**：`[[素材箱/视频日记/文件名]]` 带路径更明确；裸 `[[文件名]]` 也可。选定后归档文件名必须一致
- **改文件名后同步来源**：归档时调整了素材文件名，必须同步更新所有引用卡片
- **多来源可并列**：`[[源1]]、[[源2]]`

### 2.3 个人信息分流

发现关于青城的新认知时，同步更新 `01-关于我/` 对应文件。基于事实和原文，引用青城原话作证据（标注来源和时间），不遗漏优于不冗余。

**即使无新认知，也必须在编译后报告中显式声明**：「未发现需同步到 01-关于我/ 的个人信息」。不允许沉默跳过。

### 2.4 矛盾检测（不可跳过）

每张新卡入库前问：**这张新认知是否与某张已有卡片矛盾？** 逐张检查，逐张输出检测结果（「已检查 N 张，发现 0/X 矛盾」）。

- 认知演进（新覆盖旧）→ 新卡用 `取代：[[旧卡片]]`，旧卡末尾加注释
- 场景差异（两者都对）→ 两张卡间建立 `张力：` typed link
- 错误（旧卡是错的）→ log.md 记录后更新旧卡，保留原文 + 修正说明

不擅自删除旧卡片。

### 2.5 编译后强制同步（不可跳过）

编译完成后逐项执行，缺一不可。

#### 一、卡片数量（4 处）

| # | 文件 | 改什么 |
|---|------|--------|
| 1 | `02-知识库/index.md` | 顶部卡片数 + 日期 |
| 2 | `CLAUDE.md` | 系统架构节「N 张卡片 + M 个 MOC」 |
| 3 | `01-关于我/人物档案.md` | 知识库统计行 |
| 4 | `.claude/skills/知识库体检/SKILL.md` | 知识网络质量检测基数 |

#### 二、MOC 归入

每张新卡归入至少一个 MOC。操作：找到匹配的 MOC → 对应 `## 节标题` 下追加 `- [[卡片名]]`。归完后跑 `02-知识库/CLAUDE.md` 规定的 8 项 MOC 自检。

MOC 新增（编译产生全新主题领域）→ 同时在 `02-知识库/index.md` 的「MOC 总览」节追加新 MOC 条目，MOC 数量同步更新到第一步的 4 处数字。

#### 三、日志与调用记录

- `02-知识库/log.md`：追加编译记录（日期、来源、新增卡片数、总卡片数变化）
- `00-运转日志/知识卡片调用记录.md`：新卡追加行 `| [[卡片名]] | 0 |`

#### 四、链接类型自动分类

编译后新卡片的 `**相关链接**` 节中是裸链接。调用「typed link审计」Skill 的分类模式，逐对读取双方卡片正文，自动标注类型前缀（支撑/互见/张力/应用于），回写卡片。分类完成后验证：新卡中不应残留任何裸链接。

#### 五、网络补链

新卡片集群嵌入原有知识网络：

1. **扫描链接密度**：找出 ≤2 链的低链接卡片，补 typed link。排除 index.md 和 log.md 的反链
2. **跨集群桥梁**：新卡片与原有 wiki 中语义相关的卡片之间建立 `互见：[[xxx]]`
3. **反向链接**：从原有 wiki 卡片回链到新卡片——双向才有网络
4. **入度补链**：扫描入度=0 的卡片，从语义相关的高入度卡片补回链

硬指标：0 张 ≤2 链、0 张未归入 MOC、0 条死链。

### 2.6 归档原始文件

按来源类型移动原始文件到素材箱子文件夹。可以改文件名以更好反映内容，绝对不能修改文件内容。归档完成后 `_待处理/` 保持空。

### 2.7 阶段门控

第二阶段完成，逐项显式输出：

- [ ] 每张卡片四区完整（frontmatter + 标题正文 + 相关链接 + 来源）→ 输出「N/N 通过」
- [ ] **来源链接验证**：每张新卡的 `**来源**` 链接中指定的文件名，必须能在素材箱中找到对应文件（或确认归档时将创建）。使用缩写名/临时名/编号名的 → 阻断 → 输出「N/N 来源可解析」
- [ ] 矛盾检测已执行 → 输出「已检查 N 张，发现 0/X 矛盾」
- [ ] typed link 全为四标准类型，无自创类型 → 输出「已验证」
- [ ] source_type 在 10 种取值内 → 输出「全部合规」
- [ ] status 新卡一律 `seed` → 输出「N/N seed」
- [ ] 全库 0 张 ≤2 链卡片、0 张未归入 MOC、0 条死链 → 输出「0/0/0」
- [ ] MOC 格式统一（8 项自检通过）→ 输出「8/8 通过」

任一项不合格 → 阻断，修复后重跑。

- 更新了个人说明书的哪些部分

## 铁律

- **格式不可变**：卡片四区、MOC 节标题→卡片名、typed link 四类型、source_type 十值、status 四阶段
- **阶段顺序执行**：不跳过、不合并，每阶段门控通过后才进入下一阶段
- **只读原始文件**：不删不改素材，除非青城明确确认
- **使用优先**：每张卡片必须有可预见的调用场景
- **简单优先**：不引入额外标记、流程、手续
- **链接密度不看 index.md**：index 是行政目录，反链不计入链接密度
- **来源链接必须用素材箱实际文件名**：禁止缩写/编号/临时名。归档时改文件名必须同步更新所有引用卡片
- **门控必须显式输出**：两个门控点逐项向用户输出结果，不合格即阻断
- **Agent 分发约束**：提示词必嵌 2.1.5 全文 + 卡片格式规格；agent 只写卡片文件，协调者统一处理系统文件

## Agent 并行编译约束

> 编译通过多 agent 并行执行时生效。

**规则 1**：提示词必须逐字嵌入 2.1.5 全文（含 grep 禁令和漏卡率数据）+ 卡片格式规格（frontmatter 三字段 + 四区结构 + typed link 四类型 + source_type 十值）。禁止压缩。

**规则 2**：Agent 只写卡片文件到 `02-知识库/知识卡片/`，禁止写 log.md / index.md / MOC / 调用记录 / 人物档案。协调者统一处理。

**规则 3**：写卡前 `grep -i "{核心关键词}"` 扫描已有卡片。≥80% 语义重叠→跳过；50-80%→写入+`互见`链接。报告中列出所有跳过项。
