# 知识编译

> 版本：v3.4 · 2026-06-05

> v3.2→v3.3：对话类素材（逐字稿/线下讨论/课程演讲转写）深度整理方法升级。基于 4.18 终版与 5.30 初版对比分析，固化「亲历者复盘」整理姿势：开篇总览+重点分层+人物视角+每核心板块带边界+文末金句汇总。详细操作规范见 `03-认知工坊/对话转写深度整理提示词.md`。
> v3.1→v3.2：删除整理文档保留率硬指标（15-25%），改为「以有价值信息为判断标准，宁短勿水」。
> v2.5→v3.0：完全重构为三阶段框架。

## 总览

知识编译的本质是把外部知识拆成「以后能在某个场景被调用」的原子单元，然后嵌入到已有知识网络中。如果一张卡片找不到使用场景，它就不必存在。

整个过程分三个阶段：

```
第一阶段              第二阶段              第三阶段
素材预处理 → 深度整理  →  知识编译（卡片化）  →  全量互联与网络嵌入
                        ↓
                  个人信息同步更新（01-关于我/）
```

三个阶段顺序执行，不跳过、不合并。

### 深度整理门槛判断（前置）

在进入第一阶段前，必须先判断当前素材**是否需要进行深度整理**。深度整理不是默认流程——它是一个有触发条件的可选步骤。

**需要深度整理（强制执行）**：

| 条件 | 原因 |
|------|------|
| 多人对话/逐字稿 | 信息密度低、结构混乱，需按主题归并 |
| 单篇录音 >30 分钟 | 噪音比例高、主题散落，需手工重构 |
| 课程/演讲/分享 | 方法论密集，需「案例+原则+边界」完整结构化 |
| 无 AI 预处理的原始录音 | 直接编译会丢失结构和重点 |

**可跳过深度整理（全文通读后直接进入卡片化）**：

| 条件 | 原因 |
|------|------|
| 单人独白（视频日记 ≤10 分钟） | 已有 AI 摘要 + 原语转写，信号密度已够 |
| 已有结构化 AI 摘要的笔记 | 摘要已做第一层归并，原文干净 |
| 文章/结构化内容 | 本身已有章节和信息层级 |
| 单篇信息点 ≤5 个的高密度短文 | 直接提取即可，整理反而多一步 |

**判断流程**：扫描完素材后 → 套用上表判断 → 需要深度整理的走 1.1~1.4 完整流程 → 可跳过的直接进入第二阶段（全文通读仍然强制，在 2.1 前完成）。

## 第一阶段：素材预处理与深度整理

### 1.1 扫描素材

扫描素材来源目录，列出所有待处理文件，评估总字数和信息密度。

### 1.2 拆分判断

首先判断源文件的自然边界，再考虑字数。

**按人/按篇优先**：源文件由多个独立作者、独立分享者或独立主题的文章构成时，每人/每篇单独深度整理为一篇独立文章。不强行合并——各自有独立的脉络、风格和框架，合并只会稀释各自的完整性。

**字数辅助拆分**：在按人/按篇的基础上，单篇仍过长时再按主题拆分：

| 条件 | 处理方式 |
|------|---------|
| 单篇 < 2 万字 | 直接整理 |
| 单篇 2-5 万字 | 按主题拆为 2-3 篇 |
| 单篇 > 5 万字 | 按主题拆为多篇，每篇 1.5-2.5 万字 |

信息密度判断标准：逐字稿/闲聊类为低密度；文章/课程/方法论类为高密度。

### 1.3 深度整理

每批执行：

1. **逐文件全文深读**：每一份源文件必须用 Read 工具完整读完。不求快，逐句通读。不允许用 bash 分块预览替代，不允许用摘要替代。
2. **手工写出结构化整理文档**：基于全文深读，按主题（不是按发言顺序）归并所有方法论、框架、案例和金句。删除闲聊、重复、技术等待。保留关键原文（标注说话人）。不编造原文没有的信息。
3. **整理文档硬指标**：
   - 以有价值信息为判断标准，不设字数/保留率硬指标。宁短勿水
   - 不允许用脚本批量输出——逐句手写
   - 每个章节需有「案例 + 原则 + 边界」完整结构
   - 每篇整理文档可提取 12-25 个独立知识点

**对话类素材专项规范**（录音逐字稿、线下讨论、课程演讲转写等多人对话场景），在上述通用要求基础上追加以下要求。详细操作规范见 `03-认知工坊/对话转写深度整理提示词.md`：

- **开篇总览**：一句话来源 + "本次共N大板块" + 每板块一句话提要。让读者决定读不读、跳哪里
- **重点分层**：最有价值的洞察排第一、占最大篇幅。次要信息合并或降级为列表。不因"被讨论了很久"就给很多篇幅——讨论得久可能是因为跑题
- **人物视角**：用真实人名或角色特征替代"说话人X"，让文章有"谁做了什么"的叙事感
- **每核心板块带边界**：标注什么情况下不适用。次要板块可合并
- **文末金句汇总**：按主题分类陈列所有值得记住的判断句（用💡标记）。金句是原文中的原话，不是AI的概括
- **保留技术精度**：版本号、价格、API名称、配置参数用精确数字或表格，不用"大概""差不多"
- **整理姿势**：亲历者复盘，不是百科全书编辑。淘金——大量沙子扔掉，只留金粒

深度整理有两个目的：为知识编译提供高质量工作底稿（信号密度从 ~30% 提升到 ~90%），以及产出可对外分享的文章。

**深度整理文章的存放**：在源文件同级目录下创建 `深度整理/` 子文件夹，所有深度文章放入其中，与原始文件分开。在第二阶段知识编译完成、第三阶段归档原始文件（2.6）之前，不单独将整理文章移入素材箱子文件夹。整理文章是编译的中间产物，不是最终归档物。

### 1.4 阶段门控

每批深度整理完成后，必须自检两项：

- [ ] 该批次全部源文件已全文阅读（Read tool 逐段覆盖）
- [ ] 从知识管理和分享角度评估，整理文章合格

两项通过后进入下一批或下一阶段。不合格则重做本批。

## 第二阶段：知识编译

### 2.1 第一遍提取

逐篇阅读深度整理文档（或原文，如果跳过了深度整理），提取值得在未来被调用的信息点：

- **概念**：新概念或定义
- **判断**：值得记住的判断/结论
- **方法**：可操作的方法/步骤/框架
- **经验**：具体的经验/教训/案例
- **洞见**：反直觉或深刻的观察
- **关于青城**：关于她的新认知（分流到 01-关于我/）

一个素材包含多个独立信息点时，拆成多张卡片。每张卡片只讲一个点。

### 2.1.5 第二遍提取（强制）

第一遍提取完成后，**必须回到原文重新扫一遍**，寻找第一遍遗漏的信息点。操作方法：

1. 回到原文（或深度整理文档），逐段重读
2. 对照第一遍提取的清单，问自己：**这段里有没有第一遍没注意到的方法、判断、框架、案例、金句？**
3. 重点排查以下易遗漏类型：
   - **嵌套框架**：原文中作为"例子"出现的子方法论
   - **边界条件**：方法论的「什么情况下不适用」
   - **具体数字/公式**：量化标准、计算公式
   - **金句**：值得直接引用的原话
   - **反例**：原文提到的失败案例和教训
4. 补充提取遗漏点，与第一遍结果合并去重后进入 2.2

**铁律**：第二遍不是选做题。只提取一次会系统性遗漏约 20-30% 的可提取知识点（已验证：杨涛文章初提 10 张，二次提取补 4 张）。

### 2.2 编写卡片

按固定格式编写。格式不可变：

```markdown
---
created: YYYY-MM-DD
source_type: flomo|视频日记|口述|AI对话|课程|文章|线下分享|直播分享|微信记录|线下讨论
status: seed
---

# 标题

正文。[[双向链接]] 自然嵌入句子中。

**相关链接**：
- 支撑：[[xxx]]
- 互见：[[xxx]]

**来源**：[[原始文件名]]
```

四个固定区：**Frontmatter → 标题+正文 → 相关链接 → 来源**。顺序不可变。

**Frontmatter 三字段**：

| 字段 | 说明 | 取值 |
|------|------|------|
| `created` | 创建日期 | YYYY-MM-DD |
| `source_type` | 来源类型 | 10 种取值，见下方 |
| `status` | 成熟度 | 新卡一律 `seed` |

`source_type` 十种取值：`flomo` / `视频日记` / `口述` / `AI对话` / `课程` / `文章` / `线下分享` / `直播分享` / `微信记录` / `线下讨论`。新来源类型出现时先确认再使用，不得自创。

`status` 四阶段：`seed`（默认）→ `growing`（多来源验证）→ `evergreen`（核心反复引用）→ `dormant`（调用次数=0 且入度=0，体检 Skill 标记）。

**正文编写原则**：
- 一个卡片一个点，用自己的话精炼转述
- `[[双向链接]]` 自然嵌入句子，每张新卡至少链接一张已有卡片
- 中文写作，中文标点
- 禁止修改原始文件内容

**相关链接（Typed Link）**：

四种标准类型，强制使用，不可自创：

| 类型 | 语义 | 使用场景 |
|------|------|---------|
| `支撑` | B 在 A 的概念基础上展开 | A 提供论据/模型/框架，B 在其上构建 |
| `应用于` | A 是原则，B 是具体落地场景 | 抽象方法论→具体案例 |
| `张力` | 同一情境下两张卡给出方向不同的建议 | AI 调用时需判断适用条件 |
| `互见` | 纯语义关联，无方向性 | 跨领域关联、网络密度填充 |

`取代` 为认知演进专用（新卡片推翻旧卡片时使用），不列入常规类型。

**来源**：`[[原始文件名]]`，标注不隐藏。

### 2.3 个人信息分流

发现关于青城的新认知时，同步更新 `01-关于我/` 对应文件。更新原则：基于事实和原文，引用青城原话作为证据（标注来源和时间），不遗漏优于不冗余。

### 2.4 矛盾检测

每张新卡入库前问：**这张新认知是否与某张已有卡片矛盾？**

- 认知演进（新覆盖旧）→ 新卡用 `取代：[[旧卡片]]`，旧卡末尾加注释
- 场景差异（两者都对）→ 两张卡间建立 `张力：` typed link
- 错误（旧卡是错的）→ log.md 记录后更新旧卡，保留原文 + 修正说明

不擅自删除旧卡片。

### 2.5 编译后强制同步（不可跳过）

编译完成后必须逐项执行以下五类同步，缺一不可：

#### 一、卡片数量（4 处）

| # | 文件 | 改什么 |
|---|------|--------|
| 1 | `02-知识库/index.md` | 顶部「当前规模」卡片数 + 日期 |
| 2 | `CLAUDE.md` | 系统架构节「N 张卡片 + M 个 MOC」 |
| 3 | `01-关于我/人物档案.md` | 知识库统计行「N 张原子卡片、M 个 Skill 工作流」 |
| 4 | `.claude/skills/知识库体检/SKILL.md` | 描述中标注知识网络质量检测基数 |

新卡总数 = 旧总数 + 本次新增。一个数字改 4 个地方，一个不能漏。

#### 二、卡片归入引用（MOC）

每张新卡片必须归入至少一个 MOC。操作：找到匹配的 MOC → 对应 `## 节标题` 下追加 `- [[卡片名]]`。归完后跑 `02-知识库/CLAUDE.md` 规定的 8 项 MOC 自检。

MOC 新增（如果编译产生了全新主题领域）→ 同时在 `02-知识库/index.md` 的「MOC 总览」节追加新 MOC 条目（含典型信号和使用场景），MOC 数量同步更新到第 1 步的 4 处数字。

#### 三、调用记录（2 处）

| # | 文件 | 操作 |
|---|------|------|
| 1 | `00-运转日志/知识卡片调用记录.md` | 新卡追加行 `| [[卡片名]] | 0 |` |

#### 四、编译日志

`02-知识库/log.md` 追加编译记录：日期、来源、新增卡片数、总卡片数变化。

#### 五、查询与索引

- `知识wiki看板.md`：用 dataviewjs 动态查询，无需手动改。但编译后应打开看板确认卡片数渲染正确
- `02-知识库/index.md`：MOC 总览与 AI 检索指引在第 1、2 步已覆盖

### 2.6 归档原始文件

按来源类型移动原始文件到素材箱子文件夹。可以改文件名以更好反映内容，绝对不能修改文件内容。归档完成后 `_待处理/` 保持空。

### 2.7 阶段门控

所有深度文章的知识全量卡片化后，第二阶段结束。自检：

- [ ] 每张卡片四区完整（frontmatter + 标题正文 + 相关链接 + 来源）
- [ ] typed link 使用四标准类型，无自创类型
- [ ] source_type 在 10 种取值内
- [ ] status 新卡一律为 `seed`

## 第三阶段：全量互联与网络嵌入

### 3.1 MOC 归入

新增卡片必须归入至少一个 MOC。操作流程：

1. 判断新卡片语义归属，找到最匹配的 MOC 和子分类（`## 节标题`）
2. 在该节追加一行：`- [[卡片名]]`

MOC 格式固定：
```
## 子分类

- [[卡片名]]
- [[卡片名]]
```

无引导语、无描述（卡片名本身就是标题）、无卡片散落在节标题之外。MOC 从卡片内容自然生长，按主题聚类，不按活动/来源/事件建。命名统一 `MOC-主题`，全部放在 `02-知识库/MOC/`。

### 3.2 Wiki 语义全量互联

新卡片集群必须嵌入到原有知识网络：

1. **扫描链接密度**：找出 ≤2 链的低链接卡片，补 typed link。排除 index.md 和 log.md 的反链
2. **跨集群桥梁**：新卡片与原有 wiki 中语义相关的卡片之间建立 `互见：[[xxx]]`
3. **反向链接**：从原有 wiki 卡片回链到新卡片——双向才有网络
4. **入度补链**：扫描入度=0 的卡片，从语义相关的高入度卡片补回链

硬指标：0 张 ≤2 链、0 张未归入 MOC、0 条死链。

### 3.3 网络遍历验证

从几个关键入口卡片出发，验证 3 跳内可达的卡片数。某主题簇可达卡片 < 20 时，补跨集群桥梁。

### 3.4 协作系统维护

- **index.md**：规模数字保持最新。不在此平铺卡片——检索走 MOC
- **MOC 文档**：新增卡片归入后跑 8 项自检（见 `02-知识库/CLAUDE.md`）
- **00-运转日志/知识卡片调用记录.md**：调用记录累计更新

### 3.5 阶段门控

全量互联完成后自检：

- [ ] 全库 0 张 ≤2 链卡片
- [ ] 全库 0 张未归入 MOC
- [ ] 全库 0 条死链
- [ ] typed link 全为四标准类型（支撑/应用于/张力/互见）
- [ ] source_type 全部在 10 种取值内
- [ ] status 无越界值
- [ ] MOC 格式统一（8 项自检通过）

- 处理了哪些源文件 → 产出了几篇深度整理文章

## 铁律

- **格式不可变**：卡片四区、MOC 节标题→卡片名、typed link 四类型、source_type 十值、status 四阶段——这些都是固定格式，不因批量大小而改变
- **阶段顺序执行**：不跳过、不合并。每阶段门控通过后才进入下一阶段
- **深入挖掘**：一次性处理不了就分批，不因量大而降低提取密度
- **只读原始文件**：只引用不改动。不删素材除非青城明确确认
- **使用优先**：每张卡片必须有可预见的调用场景
- **简单优先**：不引入额外的标记、流程、手续
- **大场次必须先整理再编译**：凡逐字稿/分享/课程类素材，一律先深度整理再编译。按分享者/作者单独成篇，不强行并批
- **MOC 不是归档工具**：从内容自然生长，不按活动/来源/日期建
- **每张卡片必须归入 MOC**：没有 MOC 的卡片在调用路径上是盲区
- **链接密度不看 index.md**：index 是行政目录，它的反链不计入链接密度
- **归档移全部文件**：含课件、图片、PDF 等附属文件一并移动
- **编译后必互联**：全量互联不是可选项，是第三阶段的强制步骤
- **编译后必做五类同步**：卡片数量（4 处）→ 卡片归入（MOC）→ 调用记录（2 处）→ 编译日志（log.md）→ 查询索引验证。任何一步跳过都会导致系统数据不一致。详见 step 2.5
- **必须两遍提取**：第一遍提取后回到原文做第二遍，找遗漏的嵌套框架、边界条件、量化公式、金句、反例。只提一遍会漏 20-30%
