# 系统精简

> 版本：v1.0 · 2026-06-07

协作系统会自然膨胀——每次踩坑加一条铁律、每次升级加一段变更记录、每次加 Skill 拉长 description。不主动修剪，启动时的 token 消耗会持续涨。本 Skill 定期做一次「剪枝」。

## 审计流程

### 第一步：全量扫描

扫描并统计以下文件的体积（行数 + 字节数）：

- 根宪法 `CLAUDE.md`
- 全局 `~/.claude/CLAUDE.md`
- 所有子宪法（`*/CLAUDE.md`）
- 所有 Skill 文件（`.claude/skills/*/SKILL.md`，注意区分项目 Skill 和 dbs 等外部 Skill）
- Memory 文件（`.claude/projects/*/memory/*.md`）
- `soul.md`、`人物档案.md`
- `settings.json`、`scheduled_tasks.json`

输出一张体积排行表，标注各文件占启动加载的百分比。

### 第二步：重叠检测

**根宪法 vs 子宪法 vs Skill**：对每个领域（对话存档、知识编译、体检、框架审核等），检查同一规则是否在三处重复定义。判断标准：同一逻辑（如「6 点为界」「归档前 ls」「consolidated 标记」）是否在根宪法、子宪法、Skill 中各有一段独立表述。

**子宪法 vs Skill**：子宪法中是否有大段流程描述，而对应 Skill 已作为唯一执行体定义。

**变更记录占比**：根宪法中 changelog 行数占总行数的比例。超 20% 时标记为可迁移。

### 第三步：Memory 吸收检查

逐条读取 Memory 文件，对比根宪法和 Skill 内容，判断是否已被吸收。判断标准：
- 规则已写入根宪法行为铁律 → 可删
- 流程已写入 Skill → 可删
- 历史性改名/迁移记录 → 可删
- 描述的是已废弃机制（如双写） → 可删
- 包含 Skill/宪法未覆盖的独特信息 → 保留

### 第四步：文件分类检查

扫描所有 `CLAUDE.md` 文件，检查是否被误分类：
- 项目 spec/README（技术栈、UI 规范、已完成清单）不应命名为 CLAUDE.md
- 搁置项目的子宪法如无实际约束力，应降级或标记

### 第五步：出方案

汇总以上发现，输出精简方案，包含：
- 各文件的冗余类型和大小
- 建议动作（删除/压缩/迁移/降级）
- 预估节省（绝对值和百分比）
- 执行顺序

### 第六步：确认后执行

青城确认方案后执行，每步完成后验证。

### 第七步：验证

执行完毕后验证：
- Hook 脚本存在且可执行
- Cron 任务在运行
- 关键 Skill 软链接完整
- 检查无残留引用

## 精简原则

每次精简时遵循以下优先级：

1. **移出 > 压缩 > 删除**：变更记录迁移到独立文件优先于在行内压缩，压缩优先于删除
2. **Skill 是唯一执行体**：凡 Skill 已定义的流程，宪法只写触发条件 + 指向，不重复细节
3. **根宪法只保留 Skill 不可覆盖的元规则**：响应人格、决策权、外科手术式改动等
4. **子宪法只保留独特内容**：格式模板、参数规范、清洗规则等不在 Skill 中的操作知识
5. **Memory 不重复宪法**：已被宪法吸收的立刻删

## 铁律

- **先审计后动手**：不出方案不给青城看就动手 = 违反「先规划后执行」
- **默认不可逆操作前确认**：批量删除 Memory、重命名文件等必须先确认
- **验证不跳过**：每次精简后跑验证步骤，确保 hook/cron/Skill 正常
- **精简不是删功能**：每条被移除的规则必须在 Skill 或子宪法中有唯一执行体
