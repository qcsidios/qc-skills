# 对话存档

> 版本：v2.4 · 2026-06-18

每场对话结束时，将对话内容按约定格式存档，确保后续 AI 能快速扫描历史对话。

## 为什么是这个流程

本质是记录青城的思考演进和协作过程，不是操作日志。**仅两种触发方式**：cron 每 2h 自动触发 + 青城说「归档」「存档」。**触发来源校验（强制）**：Skill 被调后第一件事是校验来源。青城消息含触发词 OR cron → 合法；系统脚本/Hook产物/AI自作主张 → 立即终止。**6 点为界**：与日报共用时间约定。

---

## 存档流程

### 第零步：前置门禁——两次存档之间有新内容才往下走

**执行任何存档操作前，先回答一个问题：自上次存档以来，有新内容需要存吗？**

存档是存对话内容，不是存「存档动作本身」。cron 每 2h 触发一次，如果在两次 cron 触发之间没有任何实质性对话（只有 hook 输出、系统命令如 `/clear`、或空交互），则本次触发不产生新存档。

判断标准：自上次存档以来，是否产生了至少一项——
- 青城的决策（选了某个方案、定了某个方向）
- 新认知（关于他自己、关于业务、关于协作方式的新理解）
- 具体产出（代码、文档、配置变更等）

**三项都不满足 → 立即终止，跳过全部后续步骤。** 不读调用记录、不跑归档路径脚本、不加载后续内容。

典型终止场景：
- cron 触发，但两次 cron 之间只有 `/clear`、hook 输出等系统级交互
- 青城说了话但无后续回应、未形成完整对话的单轮

**判断本身必须是轻量的**——看一眼两次存档之间的对话内容即可，不需要读文件、跑脚本、调工具。门禁的目的就是避免「打开整套工具箱只为了说不用修」。

### 第一步：更新调用记录（条件执行·cron 去重）

**触发源判断**：
- **cron 触发** → 跳过此步。调用记录更新由 cron 任务自己执行，Skill 不再重复
- **手动触发（青城说「归档」）** → 按以下条件判断是否执行

执行条件：
- 青城的提问依赖他已有的认知、经验、历史判断、过往决策记录 → **执行**
- 纯技术维护、工具配置、外部调研、系统操作、闲聊 → **跳过**

判断方法：本轮对话中 AI 是否显式引用过知识卡片？引用了一张以上 → 执行。一张没引用 → 跳过。

执行时：扫描本轮对话中 AI 显式引用过的知识卡片，在 `00-运转日志/知识卡片调用记录.md` 中将对应卡片的 count +1，更新后按降序排列。

### 第二步：归档前置检查（硬性前置）

**任何归档/追加操作前，先运行归档路径脚本**：

```bash
bash AI协作工具/skills/对话存档/归档路径.sh
```

脚本自动完成：归档日计算（6点为界）→ 文件查找 → 输出目标路径和操作类型（追加/新建）。AI 直接使用脚本输出，不自己判断归档日。

- 脚本输出「追加」→ 用 `edit` 工具追加到末尾
- 脚本输出「新建」→ 用 `write_to_file` 工具新建
- 脚本提示「应考虑合并」→ 归档日有多个文件，先合并再追加

**归档日以早上 6 点为界**，由脚本强制执行。同一归档日的所有对话存档在同一文件中，不另开新文件。

### 第三步：写新建文件头部（仅新建文件时）

新建文件以「H1 标题 + 一句话总览 + AI 摘要」三段起头。一句话总览不是可选项——根目录 `运转看板.md` 的 dataviewjs 优先抓这段渲染卡片摘要，缺失则降级到"主题"字段（短而标签化）。文件名碰巧符合正则不够，正文里必须真有这一句话。

完整骨架：

```markdown
# YYYY-MM-DD 主题关键词

> 一句话总览：表面 N 件事，骨子里在做 X。

---

## AI 摘要

- **主题**：[一句话概括]
- **青城状态**：[当前状态变化，无变化可写「稳定」]
- **关键决策**：[编号列表，每条一句话]
- **产出**：[具体产出物]
- **新认知**：[关于青城的新认知或偏好变化]
- **核心引用卡片**：[[卡1]]、[[卡2]]（本次对话中作为判断核心依据的**已有**知识卡片，3-5 张。周复盘定性补充依赖此字段。**禁止填本次对话产出的新卡片**——这个字段记录的是判断依据，不是产出物）
- **待办**：[明确的后续动作，无则写「无」]
```

**一句话总览的写法**（80 字内）：
- 抓"骨子里在表达什么"——表面 N 件事，骨子里其实是一件事
- 不要复述"主题"字段——那一行已经说过一遍
- 一天只做了一件事时直接写"核心只有一句——XXX"
- 参考 5/20-5/24 已修订存档作为范例

**AI 摘要要点**：简洁，每条一行；关键决策编号，不展开；不写「可能」「或许」——只写确定的。

追加到已有文件时**只做轻量追加**：用 `---` 分隔线 + 新主题二级标题 + 正文，不重写 H1、不更新一句话总览、不更新 AI 摘要、不重编号已有主题、不合并文件、不改文件名。这些全部留给「整理修订」步骤统一处理。日常归档的目标是快速记录，不是每次追加都整理一遍。

### 第四步：追加 vs 新建

**追加到已有文件**：
- 用 `---` 分隔线
- 加新主题的二级标题（如 `## 主题 N：XXX`），**不用「七、八、九」继续编号**
- 内容紧接其下

**新建文件**：
- 文件名：`YYYY-MM-DD-[主题关键词].md`
- 写完头部三段（H1 + 一句话总览 + AI 摘要）后开始正文

### 第五步：写对话记录（按主题模块组织）

**核心原则：记录的是思考，不是操作。**

#### 写作要求

- **以青城的思考为主线**：他在想什么、想干什么、意图是什么、想法怎么演变的
- **写决策路径，不写操作清单**：不是「青城说了 X → 知白做了 Y」，而是「青城想要达到什么效果、为什么选择了这个方案、中间遇到了什么问题、怎么解决的」
- **捕捉协作模式**：他是怎么用我的、我哪里做得好哪里犯了错、他对我的行为做了什么修正
- **按主题合并，不按时间线分节**：同一主题的多个来回归入同一节

#### 写法示范

- ✅ 「青城对归档频率经历了完整的探索：10min→30min→1h→整点→最终30min。背后是一个发现过程：太频繁没必要，整点有盲区（对话中 cron 不触发）。整个过程体现了他对系统设计的敏感——不是随便调参数，而是在找平衡点。」
- ❌ 「青城将自动存档定时从 10 分钟改为 30 分钟。知白删除旧 cron 任务，新建 30 分钟 durable cron。」
- ❌ 「任务1：扫描今日存档文件，补充标记。任务2：检查调用记录，无变动。任务3：生成日报。」——这是操作清单，不是思考记录。应该写：为什么做这些判断、跳过某步骤的依据是什么、过程中发现了什么问题

#### 不保留的

- 纯执行确认和工具调用过程
- 重复操作细节
- 没有新认知或决策的闲聊

### 第六步：信息回流

存档完成后检查：
- 对话中是否产生了应进入知识库的新认知？→ 编译为原子卡片写入 `02-知识库/`
- 对话中是否更新了对青城的认知？→ 更新 `01-关于我/` 对应文件
- 回流时遵循正常编译流程，不因「对话刚结束」而跳过深度处理

判断标准：回流的是知识/认知，不是对话本身。对话存档是原始记录，知识库是编译后的成果。两者互补不重复。

**重要：日常存档不加 `consolidated: true`。** 存档流程（第一步到第六步）只管写入/追加，不碰 frontmatter 的 `consolidated` 字段。这个标记是「整理修订」步骤的幂等机制，只有6点后hook自检或当天最后一次归档时才写入。日常存档中添加此标记 = 滥用，会导致第二天hook误判为已整理而跳过。

### 存档自检清单（日常归档·轻量）

写完存档内容后、告诉青城「完成」之前，逐项检查：

- [ ] **第零步门禁已通过**：本轮对话确实有实质性内容（决策/认知/产出），不是纯系统操作
- [ ] 运行过 `对话存档/归档路径.sh`，确认归档日期和追加/新建
- [ ] 同一归档日只有 1 个文件（没有意外新建了第二个）
- [ ] 第一步条件判断正确：手动触发 + 涉及知识 Wiki 才跑调用记录更新；cron 触发跳过
- [ ] 写的是思考和决策路径，不是操作流水账
- [ ] 核心引用卡片是**已有**卡片，不是本次产出的新卡片（无引用时填「无」）
- [ ] 只做了轻量追加（分隔线+主题+正文），没有重写总览/摘要/重编号/改文件名
- [ ] 没有在日常存档中添加 `consolidated: true`

**任何一项未通过 → 修正后再报完成。**

> 主题编号连续、AI摘要覆盖全部主题、文件名反映全天内容——这些是「整理修订」的职责，日常归档不处理。

## 整理修订（一天结束时的精简动作）

当一天的对话存档因为多次追加变得碎片化时，做整理修订让它变成可读的整体。

### 触发时机

**强制触发（必跑）**：
- **每日最后一次归档时**：青城说睡了 / 明确表示当天结束
- **6 点之后启动 Claude Code 时由 hook 自检 + AI 直接执行**：hook 检查昨日对话存档是否带 frontmatter `consolidated: true`，缺标记则注入"待执行"指令，AI 收到就**直接整理修订并补标记**，不要询问青城是否触发；不要把"今天开了第几个终端"当判断依据——只看标记。已有标记则 hook 静默，AI 不要主动跑

**幂等机制**：整理修订完成后必须在文件 frontmatter 写入 `consolidated: true` 和 `consolidated_at: YYYY-MM-DD`。这是下次 hook 判断是否需要再跑的唯一依据。

**条件触发**：
- 青城说「整理一下」「重新整理这天的存档」
- 当天文件被追加 3 次以上
- 章节编号出现重复（如同一文件里两个「一、二、三」）
- 反思在多处重复（如同一错误的反思在 3 处分别写过）

**触发后必做**：① 重新组织主题 ② 合并冗余 ③ 提炼跨主题隐线 ④ **同步修改文件名**让标题反映全天内容（而不是只反映上午或某个阶段的事件）

### 整理动作

1. **重命名文件**：让标题反映全天内容，而不是只反映上午（如「日报改名」→「基础设施重构与知识库 MOC 化」）
2. **用主题模块组织**：`## 主题 1：XXX` / `## 主题 2：XXX`，**不用「一二三四五六」硬章节号**
3. **合并重复反思**：同一规则的反复失败 → 表格 + 元层级反思
4. **提炼跨主题隐线**：找出当天所有主题在表达的「同一件事」，作为最后一段
5. **合并产出清单**：按类别分组（协议/结构/工具链/Skill/知识库/个人说明书），不按时段切
6. **写入幂等标记（不可省）**：整理修订完成后，在文件顶部加 frontmatter `consolidated: true` 和 `consolidated_at: YYYY-MM-DD`。漏写这一步 = 下次启动 hook 还会再报"未整理"，等于工作做了一半。
7. **行数检查（整理后独立执行）**：整理修订完成后，检查文件行数。超过 400 行 → 触发二次浓缩（合并相似主题、删除已被知识库承接的重复认知、把过程性来回归并成结论）。未超 400 行 → 跳过压缩，不额外操作。**整理和压缩是两个独立步骤，不要混在一起做。**

---

## 铁律

- **先判断再动手，没内容不铺工具**：加载 Skill 后第一件事是看本轮对话有没有实质内容。没有→立即终止，不往下读文件、不跑脚本。门禁本身是轻量的，一眼判断即可
- **日常归档只做轻量追加**：分隔线+主题+正文，不重写总览/摘要/重编号/改文件名。全部留给整理修订
- **同一归档日只留一个文件**：ls 当天文件夹时如果已有文件，必须追加不得新建。意外产生多个文件时立即合并
- **新建文件必含一句话总览**：运转看板 dataviewjs 第一优先级抓这段；缺失则降级渲染
- **按主题归并，不按时间分节**：同一主题的多个来回归入同一节。用 `## 主题 N：XXX` 而非「一二三四五六」
- **整理修订幂等触发**：每日最后一次归档 OR 6 点后 hook 注入时 AI 直接执行。完成必写 `consolidated: true`
- **跨主题隐线必提炼**：找出当天所有主题在表达的同一件事
- **信息不重复沉淀**：对话存档 + 知识库/说明书各司其职，同步回流但不复制。中文文件名
